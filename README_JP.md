## 🎯 VectFox とは？

優れた VectHare をベースに構築された VectFoxは、**SillyTavern 向けの高性能長期記憶システム**です。単なる多言語フォークをはるかに超えた存在です — すべての高度な検索ロジックは本物のベクトルデータベース（Qdrant）のサーバーサイドで動作し、構造化されたイベントベースのアプローチが生の要約チャンクに取って代わり、劇的に高精度なリコールを実現します。2,000 件以上のメッセージでも 3 秒以内に結果を返します。ネイティブ多言語対応：英語・日本語・韓国語・繁体字中国語・簡体字中国語。

元の VectHare をフォークした理由は、私自身の [MVU Game Maker](https://github.com/KritBlade/MVU_Game_Maker) プロジェクトで、とても大きな規模のデータを扱う必要があったからです。こうしたプロジェクトには、次のような特徴があります：
- **非常に大きな規模：1 つのストーリーに 2,000 件以上の返信、1 返信あたり 1,000 文字以上。要約検索の応答は 3 秒未満**
- 英語以外の言語にも対応（日本語、韓国語、繁体字/簡体字中国語）。英語はデフォルトで対応しています。
- [MVU Game Maker](https://github.com/KritBlade/MVU_Game_Maker) の機能タグをすべて自動で除去します。
- 本当に使える超長期記憶（2000+ メッセージ）

一般的な SillyTavern のメモリ拡張は、この規模の負荷になると完全に耐えられません。特に、ストーリー内に [MVU Game Maker](https://github.com/KritBlade/MVU_Game_Maker) が使う機能タグが大量に含まれている場合はなおさらです。これらのタグは、記憶検索にはまったく役に立ちません。だから私は、大規模なデータでもこうしたタグをきれいに取り除きつつ、高速なベクトル化を維持できる仕組みが必要でした。

多くのメモリ拡張は、100 メッセージ以下のチャットを前提に作られています。そのくらいの規模なら、たしかによく動きます。でもチャットがそのラインを超えて伸びていくと、古いメッセージをどんどん強く圧縮せざるを得なくなります。最終的には、最近の履歴だけは細かく残っているのに、古い内容はひどく圧縮されてぼんやりしたものになります。しかも、この問題にはほとんど逃げ道がありません。100+ メッセージ分の生のコンテキストを、プロンプトや自動生成されたロアブックエントリにそのまま詰め込むことはできないからです。古い記憶は**圧縮せざるを得ない**。そして圧縮すれば、細部は失われます。

四角い車輪の車は、どれだけ調整しても快適には走りません。必要なのは、この用途に合った正しい道具です。私が本当に必要としていたのは、すべての記憶をきちんと保存できる、専用のベクトルデータベースバックエンドでした。

***私は SillyTavern のために、アーキテクチャ的に正しく、プロダクションに近い設計のメモリシステムを構築することにしました。メモリをハードコアにしよう！***

そこで VectFox は、専用のベクトルデータベースを使って、チャット内の**意味のあるすべてのイベント**を保存します。最初のメッセージでも、2,000 番目のメッセージでも、意味のあるイベントはすべてデータベースに残り、SillyTavern がいつでも検索できる状態になります。私が作りたかったのは、10k+ メッセージまでスケールでき、検索の往復が数秒以内に終わる、SillyTavern 向けの本番品質のメモリベクトルシステムです。

### どんな問題を解決するのか

- 🧠 **元の VectHare は「何を覚えておくべきか」を考えてくれません。** VectFox は、その上に LLM 駆動の **EventBase** 抽出レイヤーを追加します：AI が「意味のあるイベント」を自分で判断し、各イベントに登場人物・アイテム・場所・概念のタグを付けます。
- 🤖 **元の VectHare はあなたの質問を推論しません** — 表面的なテキスト類似度だけで照合します。VectFox はオプションの **Agent Mode** を追加し、小型の LLM で多角的な検索を計画することで、生のクエリでは見つからなかった記憶も拾い上げます。
- 😩 記憶として保存する前に、[MVU Game Maker](https://github.com/KritBlade/MVU_Game_Maker) が使う機能タグをすべて除去します。
- 🧠 [MVU Game Maker](https://github.com/KritBlade/MVU_Game_Maker) のキャラクター記憶に加えて、ストーリーをベースにした記憶を追加します。
- 💸 長い会話で、関係のない履歴が token 予算を圧迫してしまう問題を減らします。
- ✍️ 重要なイベントをキャラクターに思い出させるために手作業でコンテキストを編集する必要がなくなります。
- 🎯 **ゴミを入れればゴミが出る。** 生のチャットを非構造化されたテキストの塊にまとめてベクトル化すると、ノイズが多く、シグナルの低い検索結果になります。EventBase は**厳密な構造化フォーマット**（`characters`・`items`・`locations`・`concepts`・`keywords`・`importance`・`DateTime`）を強制し、ベクトル DB にネイティブな形式で保存します—単なるテキストチャンクではありません。Agent Mode はその高度に構造化されたデータを**ターゲット構造化クエリ**（キャラクター・概念・場所によるフィルタリング）で検索することで、非構造化された要約に対する単純なテキスト類似度検索と比べて、ヒット率を劇的に向上させます。

**VectFox の解決策：** **Qdrant** を専用ベクトルデータベースとして使い、チャットがどれだけ長くなっても、意味のあるイベントをすべて保存します。まだ追加サービスを動かす準備ができていないユーザー向けには、**A1 と A2 パスだけを使う「軽量版」** も用意しています。これは SillyTavern 組み込みのベクトルストア上で動くため、追加ソフトウェアは不要です——フルのベクトル DB 版と多くの機能を共有しつつ、扱える規模だけが小さめです。本格的な長期記憶システムを作る準備ができたら、Qdrant を使う **A3 パス**へアップグレードしてください。

注意：Qdrant は無料でオープンソースです。

### VectFox が解決しようとしていないもの

VectFox は**記憶**システムであり、トラッカーではありません。クエスト進行・キャラクターのステータス・ライブな世界状態の追跡は行いません。これらが必要な場合は、[MVU Game Maker](https://github.com/KritBlade/MVU_Game_Maker) と組み合わせて使ってください — クエスト・キャラクター・ステータス管理用の GUI を備えた、キャラクター中心のトラッキングシステムです。両方を組み合わせて運用すれば、長編 SillyTavern ロールプレイで悩まされる記憶と状態に関する問題のおよそ 90% をカバーできます。

## 🧠 どう動くのか

ベクトル検索は、とても賢い「検索」機能のようなものです。完全に同じ単語を探すのではなく、**意味**を探します。たとえば「お腹が空いた」と入力すると、「ランチに行こう」というメッセージも見つけられます。*意味*が近いからです。

VectFox は、互いに連携する 2 つの中心的な考え方で作られています：

### 1. EventBase — 返信ごとの要約ではなく、時間ウィンドウからイベントを抽出する

多くのメモリ拡張は、AI の返信 1 件（または数件）を、ひとまとまりのテキストに要約します。聞いただけだと自然に思えますが、実際に返信の中身を見てみると問題が見えてきます：

- 100 文ある返信の中に、**5 つの意味あるイベント**（戦闘、発見、約束、アイテム交換、関係性の転機）が含まれているかもしれません。ただし、それらは 95 文分の会話のつなぎ、場面描写、雑談の中に埋もれています。
- 別の返信には、**イベントが 0 個**かもしれません——ただの雑談だけです。
- さらに別の返信では、1000 文字を使って**1 つのイベント**だけを描いているかもしれません。

「返信ごとに要約する」方式では、この 3 つのケースがすべて「返信 1 件につき 1 つの塊」に押しつぶされます。イベントの境界が失われ、重要な転機と埋め草が混ざり、実際に何が起きたかに関係なく、同じ形のデータになってしまいます。

**EventBase は、最近のメッセージで作られる時間ウィンドウを見て、実際に起きた内容に応じて 0 個、1 個、または複数の構造化イベントを抽出します。** 各イベントは独立したレコードで、豊富なメタデータ（`characters`、`locations`、`items`、`concepts`、`importance`、`DateTime` など）を持ちます——ただの説明文ではありません。

たとえば Tav と Astarion が、他の 3 人の仲間や背景ストーリーのノイズに囲まれながら、100 文ほどの長い買い物シーンを話していたとします。EventBase は、その大量のテキストから次のようなイベントを 1 つ抽出するかもしれません：

```
{
event_type:    item_acquired
importance:    6
text:          タヴとアスタリオンはバルダーズ・ゲートで鎧の買い物をした。アスタリオンは値段をからかった。
               タヴは革製の胸当てを80gpで購入した。
DateTime:      1492-08-15T14:00:00
cause:         タヴはシャーの試練の遠征前に、より良い鎧が必要だった
result:        タヴは今、革製の胸当てを装備している。パーティーの資金から80gpが使われた
characters:    [タヴ, アスタリオン]
locations:     [バルダーズ・ゲート, ソーサラス・サンドリース地区]
factions:      []
items:         [革製の胸当て, 80gp]
concepts:      [鎧の買い物, パーティーの経済]
keywords:      [鎧, 革, 胸当て, ゴールド, 買い物]
open_threads:  [シャーの試練の準備]

}
```

あとで「あの買い物のこと覚えてる？」と聞いたとき、VectFox が検索するのは**このイベント**です。100 文の生テキストでも、買い物シーンと周囲の無関係な雑談を平均化したぼんやりした要約でもありません。

### 2. 専用ベクトル DB が自然な選択になる理由

返信ごとの要約だけをするなら、実はベクトルデータベースは**必要ありません**。作られるのは、だいたい返信 1 件につき 1 つの塊で、形もほぼ一定のデータです。チャットが伸びるたびに、それを再帰的に再要約していくだけです。この用途なら、単純なテキストファイルでも足ります——多くの古いメモリ拡張が本物の DB を使わなかったのも、そのためです。

でも EventBase を採用すると、状況はまったく変わります：

- 2,000 件の返信 → **1,000～3,000 個の構造化イベント**が生まれる可能性があります（複数イベントを抽出する返信もあれば、ゼロの返信もあります）
- 各イベントは豊富なフィールドを持ちます：characters、locations、items、concepts、keywords、importance、timestamps
- ユーザーが会話している最中に、**意味 + キーワード + メタデータフィルタリング**で、関連する 5～10 個のイベントをリアルタイムに見つける必要があります

これはまさに、**Qdrant のような専用ベクトルデータベースが得意とするワークロード**です。構造化フィールドを持つ小さなレコードが大量にあり、密ベクトル類似度、スパースなキーワード検索、メタデータフィルタリング、さらにコーパス全体に対するグローバル BM25 重み付けが必要になります。これをフラットな要約ファイルでやろうとすると、線形スキャンになり、キーワードインデックスもなく、メタデータフィルタもなく、数百エントリを超えるとスケールしません。

EventBase は Qdrant の使用を*強制*しているわけではありません——A1/A2 の軽量パスは SillyTavern 組み込みの Vectra で動きます。ただ、チャットイベントが数百を超えてくると、このデータ形状に本当に合ったストレージレイヤーは Qdrant です。

### 3. Agent Mode — AI に検索を計画させる（オプション、Qdrant / A3 のみ）

純粋なベクトル検索には弱点が一つあります：あなたが文字通り入力したものしか検索できないのです。「身代金を払った理由を覚えてる？」と聞けば、検索はその言葉に一致するイベントを探します。しかし本当の答えは、身代金につながった誘拐、金額が決まった交渉、そしてその後のキャラクターの感情的な反応に関わっているかもしれません——そして、あなたの生の質問にはそれらのどれも含まれていません。

**Agent Mode は小型の「プランナー」LLM を追加します**。最近のチャットと意味的に最も近いマッチを読み、こう考えます：*「この質問にきちんと答えるには、他にどの角度から検索すべきか？」* 1〜4 個のフォローアップクエリを出力し、Qdrant に並列で投げます。

**具体例。** あなたが入力します：
> *アスタリオンに言う：「ガントレットで何を考えていたの？」*

Agent Mode なしでは、検索は次を見つけます：
- ✓ ガントレット探索イベントそのもの（「ガントレット」の直接一致）

Agent Mode を使うと、プランナーは自動的に次の角度を追加します（このクエリに対するプランナー出力の例）：

```json
{
  "queries": [
    "ガントレット 探索 入り口",
    "アスタリオン 試練 反応",
    "影界 発見 物語"
  ],
  "filters": {
    "characters_any": ["Astarion"],
    "concepts_any": ["ガントレット", "試練"]
  }
}
```

3 つの並列 Qdrant 検索が返すもの：
- ✓ ガントレット探索シーン（直接）
- ✓ アスタリオンが試練に反応した瞬間（感情的な背景）
- ✓ 影界での発見と発言（関連する物語の節目）

すべての結果は元の検索結果とマージされ、同じ 4 重み再ランカーを通ります。メイン応答 LLM は瞬間的なシーンだけでなく、**完全な物語の流れと感情の背景**を受け取ることになります。

**Agent Mode が A3（Qdrant）と組み合わさる理由**：
- プランナーの各クエリは独立した Qdrant 呼び出しです。Qdrant は 4 つの並列クエリを 1〜3 秒で処理します。
- AgentMode は Qdrant が必要です——Standard バックエンドでは完全にスキップし（`mode=SKIPPED reason=requires_qdrant_backend` をログ出力）、プリサーチをそのまま返します。優雅なフォールバックはなく、設計上の意図です。
- Qdrant のペイロードフィルタ（`characters_any`、`locations_any`、`factions_any`、`items_any`、`concepts_any`、`event_type_any`、`importance_gte`）により、プランナーは各検索を正確に絞り込めます——標準バックエンドはこれを公開していません。

**コストとレイテンシ**：プランナーに GPT-4o-mini や Haiku などの小型モデルを使う場合、1 ターンあたり約 $0.0002、追加で約 2〜5 秒。**純粋に加算的**で、通常の検索を置き換えることはなく、プランナーが失敗すれば標準フローに自動的にフォールバックします。専用の **AgentMode** タブで設定。デフォルトはオフ。

> 💡 Agent Mode が最も活きるのは、**反省的な質問**（「なぜ……？」「……を覚えてる？」）、**曖昧な呼び戻し**（「父親についてのあの件」）、そして**因果連鎖のクエリ**——つまり、文字通りのユーザーメッセージに検索アンカーが含まれていない場合です。メッセージにすでに正しいキーワードが含まれているような直接的な検索では、標準の事前検索だけで大部分の仕事をしてくれます。その場合、Agent Mode は仕上げの磨きのようなものです。

### 🧠 従来のメモリ拡張より優れている理由

既存のメモリ拡張の多くは、次の 2 つのアプローチのどちらかを使っています。どちらも、チャットが長くなるにつれて細部を失います。理由と、EventBase がどう避けているかを見てみましょう：

| 観点 | 📝 ローリング要約 <br>*（多くの「メモリ」拡張）* | ✂️ 生チャンク <br>*（古いベクトル RAG）* | 🧬 EventBase <br>*（VectFox）* |
|---|---|---|---|
| **何を保存するか** | どんどん長くなる 1 本の要約テキスト | 各メッセージを切った生チャンク | メタデータ付きの構造化イベントレコード |
| **メッセージ 100 時点** | ほぼ完全 | 完全 | 完全 |
| **メッセージ 200 時点** | 大きく圧縮——名前、数字、一度きりの細部がずれたり消えたりする | token 予算があふれる——古いチャンクはスコアで削られるか捨てられる | **完全**——古いイベントはまだ DB にあり、関連すると浮上する |
| **メッセージ 1,000+ 時点** | 実質的にはぼんやりした塊 | DB が肥大化する；生チャンクは信号が弱いので検索がノイズだらけになる | **完全**——現在のシーンに関係する少数のイベントだけが取り出される |
| **「圧縮」で何が起きるか** | 要約をさらに要約するため、毎回情報が失われる | 圧縮はしない——ただし統合もしない；生テキスト検索は当たり外れが大きい | 一度だけ、意味ベースで処理——*意味のあるイベント*を抽出し、埋め草を捨てる。イベント自体の細部は残る。 |
| **検索信号** | なし——要約全体が毎回注入される | 生テキストへのベクトル類似度（言い換えは拾えるが、ノイズも多い） | 豊富なフィールド（`characters`、`items`、`locations`、`concepts`、`keywords` と密な意味）に対するベクトル + BM25 ハイブリッド |
| **細部はどこへ行くか** | 圧縮された時点で永遠に失われる | チャンクのスコアが閾値を下回ると失われる | **どこにも行かない**——イベントはベクトル DB に残り、関連すると浮上する |
| **何を注入するか** | 動作中の要約全体（毎ターン、毎回） | 意味的に近い数件の生メッセージ | 今のメッセージに関係するイベントだけ |

**大事なポイント：** ローリング要約が細部を失うのは、スペースを空けるために古い内容を*捨てる*からです。生チャンクが細部を失うのは、*規模が大きくなると検索が壊れる*からです。EventBase は意味のあるイベントをすべて保持し続けます——そして、ベクトル + キーワード検索が「今 AI に見せる価値がある 5～10 個」を選びます。細部は圧縮されません。**無関係なものだけがフィルタされます**。

> 💡 **メッセージの言い方は、検索結果に大きく影響します。** 検索はあなたの返信テキストを手がかりに動くため、どんな言葉を使うかが重要です。たとえば、*「Mayla、私がなぜ身代金を払ったか覚えてる？」* と *「Mayla、私がなぜ 2000 ドル払ったか覚えてる？」* では、返ってくるイベントがかなり変わります。「身代金」は、そのストーリーラインに関係するイベント（誘拐、交渉、受け渡し）をまとめて引き出します。一方で「2000 ドル」は、主に 2000 という数字が文字通り出てくるイベントにだけマッチします。AI に特定のシーンを思い出させたいなら、正確な数字のような付随的な情報ではなく、そのシーンで**物語上意味のある言葉**を使ってメッセージを固定してください。
>
> 1,500 イベント規模のチャットで横並びテストをしたところ、A3（Qdrant）は適切に固定されたクエリで身代金イベントを **#1 / #2** にランク付けし、数字中心のクエリでも上位に浮上させました。A1 / A2（標準バックエンド）も**同じ身代金イベントを見つけてはいた**ものの、ランクは低めでした（適切に固定されたクエリでもおおむね **#3 / #5** あたり）。これは、実際にプロンプトに注入される上位のイベント枠を外れることが多い順位です。差は構造的なもので——A3 は疎なキーワードインデックスを使って完全コーパスを検索しますが、A1 / A2 は密ベクトル層がたまたま拾った候補だけを採点します（下のパス比較を参照）。固定語はどのバックエンドでも重要ですが、A3 は外したときにより寛容です。

## 🔍 ハイブリッド検索：A1、A2、A3 パス

VectFox は、最適な結果を見つけるために**2 種類の信号**を組み合わせます：

- **信号 1 - ベクトル類似度** — 意味ベース（「お腹空いた」が「ランチに行こう」にマッチする）
- **信号 2 - BM25 キーワードスコア** — 完全な単語一致（「Astarion」が「Astarion」にマッチする）

バックエンドと設定に応じて、この 2 つの信号を組み合わせる**3 つのパス**があります：（あなたのブラウザから、docker 上の専用ベクトルデータベースまで）

### A1 — Standard バックエンド + BM25
ブラウザがベクトル検索を実行して上位 ~100 件の候補を取得し、その候補に対してだけ BM25 キーワードスコアを計算します。最後はシンプルな加重和です：`α × vectorScore + β × bm25Score`。

**トレードオフ：** 軽くて速く、遅めのコンピュータにも向いています。ただし、完璧なキーワード一致が上位 100 件のベクトル結果に入っていなければ、完全に見えなくなります。

### A2 — Standard バックエンド + Hybrid（A3 パスを使わない多くのユーザーにおすすめ）
ブラウザがベクトル検索を実行して上位 ~100 件の候補を取得し、**同じ候補プール**を 2 通りでランク付けします——ベクトル類似度で一度、BM25 で一度——そして 2 つのランクリストを次の方法で融合します：
- **RRF（Reciprocal Rank Fusion、逆順位融合）** — 生のスコアではなく、各リスト内の*順位*を使って結果を結合します
- **デュアル信号ボーナス** — 両方のリストに出てきた結果には最大 +8% のブースト、片方だけに出てきた結果には小さなペナルティ（ベクトルのみ ×0.55 / テキストのみ ×0.60）

> ⚠️ **A2 は独立した疎検索ではありません。** 両方のランクリストとも*同じ密 ANN 候補プール*から取られます。BM25 は密レイヤーがすでに返した候補を並べ替えることはできますが、密 ANN が見逃したキーワード単独一致を浮上させることはできません。各イベントに本物の疎ベクトルを保存して全コーパスに対して疎検索を実行するのは **A3** だけです。

**例：** 「Astarion drinks blood.」で検索した場合、密 ANN が Astarion と血に関するイベントを返したならば（例えば「吸血鬼/空腹」が意味的に近いため）、両方のランキングがそれを浮上させ、デュアル信号ボーナスがさらに押し上げます。希少なキーワード単独イベントが上位 100 件の窓の外にある場合、A2 では見つかりません——A3 なら見つけます。

**トレードオフ：** 速めのブラウザ環境ではより良い融合結果が得られますが、リコールは依然として密 top-K（~100 件候補）に制限されます。

### A3 — Qdrant ネイティブスパース + サーバーサイド RRF + フォーミュラ再ランク（最高精度）
A3 では**すべてが Qdrant 内部の単一 API 呼び出しで実行**されます。A1 / A2 に対する真の構造的アドバンテージは「速い」だけではありません——**A3 はキーワードでも完全コーパスを実際に検索する**のに対し、A1 / A2 は密ベクトルでしか検索しないという点です：

1. **完全コーパスに対する密 + 疎ハイブリッド検索** — Qdrant は upsert 時点でイベントごとに疎キーワードベクトルを保存します。クエリ時には密インデックス（意味）と疎インデックス（キーワード）が**コレクション内のすべてのイベントに対して並列に検索**され、ネイティブ RRF で 2 つの結果リストが融合されます。これこそが A3 が本当にスケールする理由です：稀なキーワードが 1,500 件前の一つの目立たないイベントにしか現れない場合でも、疎インデックスは直接それを見つけます——密ベクトル層がたまたまそれを浮上させるかどうかには依存しません。（A1 / A2 は密 ANN が返した top-K 候補にしか採点しないため、その窓の外にあるイベントはどれだけキーワードが一致しても見えません。）BM25 IDF もサーバー側でグローバルに計算されるため、稀な単語のスコアは構造的に正しくなります。

2. **サーバーサイド・フォーミュラ再ランク** — Qdrant は次に、同一呼び出し内でハイブリッド結果に 4 重みの再ランク式を適用します。各 `w_X` はユーザー調整可能な重みスライダーで、乗算される項は重みが綺麗に合成されるよう正規化されています：`w_cosine × RRF_score + w_importance × (importance/10) + w_persist × (1 if should_persist else 0) + w_recency × exp_decay(source_window_end → chatLength)`。最終ランキング済みリストはすでに並び替えられた状態で返ってきます。追加の往復もなく、ブラウザの JavaScript もスコアリングを行いません。

3. **サーバーサイド・フィルタリング** — 最低重要度の閾値とコンテキスト重複排除のカットオフ（最近のチャットにすでに表示されているイベント）は、結果が返ってきてからではなく、Qdrant の内部で適用されます。閾値を下回るイベントはサーバーを出ることすらありません。

4. **AgentMode セマンティック事前フィルタリング + 多角度クエリ** — AgentMode は互いを強化し合う 2 つのことを行います。まず、プランナー LLM がユーザーの質問を複数のサブクエリに分解します——表面的な意味だけでなく、*これはどのように起きたのか？*、*その前に何があったのか？*、*結果はどうなったのか？*、*他に誰が関与していたのか？* も含め、各角度が独立したベクトル検索として実行されます。次に、プランナーが構造化されたエンティティフィルター（`characters_any`、`locations_any`、`factions_any`、`items_any`、`concepts_any`、`event_type_any`、`importance_gte`）を発行し、同じ呼び出し内で Qdrant ペイロード節として適用され、ベクトル検索が実行される*前に*候補プールを絞り込みます。この組み合わせは強力です：多角度クエリが広範なセマンティックネットを張り、ピンポイントフィルターが各検索を正しいエンティティにスコープされるよう保証します——無関係なキャラクター、場所、イベントタイプがリコールスロットを奪うことはありません。

ブラウザが担当するのは、アンカーブースト（フレーズマッチング）、ペアワイズ重複排除、そして複数コレクションにわたる最終マージのみです。

**例：** 「I cast Fireball at the dragon.」で検索すると、Qdrant は密（呪文/攻撃の意味）と疎（文字通りの「Fireball」+「dragon」）を同時に検索し、RRF で融合し、importance/recency フォーミュラで順位付けし、低重要度のイベントを除外してから——最終的な注入準備済みリストを 1 回の呼び出しで返します。

**AgentMode 例：** 「Shadowheart とどんな取引をしたのか？」と尋ねると、プランナーが `characters_any: ["Shadowheart"]` を発行し、複数のサブクエリに展開します：元の質問、*「Shadowheart に関わる合意や約束」*、*「Shadowheart との取引につながったイベント」*、*「Shadowheart が見返りに要求したこと」*。各サブクエリは Shadowheart タグのイベントにすでに絞り込まれた Qdrant 候補プールに対して実行されます——広範なセマンティックリコール、クロスキャラクターのノイズなし。

**トレードオフ：** 精度は最高で、大規模データでは速度も最速です。ただし、Qdrant インスタンスが必要です（無料・オープンソース）。→ [Qdrant インストールガイド](Doc/Qdrant_install.md)

## ✨ 機能

### 🧬 EventBase — LLM が抽出するチャット記憶
LLM がチャットを構造化イベントとして要約し、重要度/最近度/持続性の重みを付けます。4 重みの再ランカーが、どのイベントを注入するかを決めます。内蔵の重複抑制により、最近のメッセージ内ですでに見えているイベントは抑制されます。

各イベントは生テキストではなく、構造化されたレコードです。以下は、LLM が Astarion の買い物ウィンドウから生成する例です：

```
event_type:    item_acquired
importance:    6
text:          タヴとアスタリオンはバルダーズ・ゲートで鎧の買い物をした。アスタリオンは値段をからかった。
               タヴは革製の胸当てを80gpで購入した。
DateTime:      1492-08-15T14:00:00
cause:         タヴはシャーの試練の遠征前に、より良い鎧が必要だった
result:        タヴは今、革製の胸当てを装備している。パーティーの資金から80gpが使われた
characters:    [タヴ, アスタリオン]
locations:     [バルダーズ・ゲート, ソーサラス・サンドリース地区]
factions:      []
items:         [革製の胸当て, 80gp]
concepts:      [鎧の買い物, パーティーの経済]
keywords:      [鎧, 革, 胸当て, ゴールド, 買い物]
open_threads:  [シャーの試練の準備]
```

2 つの信号（意味 + キーワード）は、この豊富なフィールドセット全体に対して動作します。そのため、「armor for the dungeon」のようなクエリは concepts/open_threads 経由でヒットし、「Astarion 80gp」は characters/items/keywords 経由でヒットします。この構造は Qdrant ベクトルデータベースに自然に合う形式なので、他のどんなメモリ拡張よりも高いヒット率を実現できます。

### 🤖 Agent Mode — LLM が計画する多角的検索（Qdrant / A3 のみ）
小型のプランナー LLM が最近のチャットと事前検索の上位候補を読み、ユーザーの質問の異なる角度を狙った 1〜4 個のフォローアップ Qdrant クエリを出力します。結果は並列に取得され、同じ 4 重み再ランカーを通って統合されます——**純粋に加算的で、通常の検索を置き換えることはありません**。どんな失敗時でも標準フローに自動でフォールバックします。文字通りのユーザーメッセージに検索アンカーが含まれない**反省的な質問、曖昧な呼び戻し、因果連鎖クエリ**で最も効果を発揮します。

- **プロバイダー/モデルの継承** — AgentMode タブで空欄のままにすれば、要約器の設定が継承されます。プランナーのコストを下げるために、より安価なモデル（例：Haiku 4.5、GPT-4o-mini）で上書きすることもできます。
- **言語マッチングのプロンプト** — プランナーはチャットの言語（中国語、日本語、韓国語、ラテン文字、英語）でクエリを出力し、固有名詞は元の形を保ちます。クロス言語の混入はありません。
- **調整可能なスライダー** — プランナーに送る過去ターン数（1〜10）、プランナーに見せる候補数（5〜20）、最大クエリ数（1〜4）、タイムアウト（1〜60 秒）、デバッグログ。
- **実際のコスト** — 小型で高速なモデルを使えば 1 ターンあたり約 $0.0002、追加レイテンシは 2〜5 秒。デフォルトはオフ——長編の呼び戻しが必要なときに有効化してください。

設定の **AgentMode** タブ、または上の「どう動くのか → Agent Mode」セクションで、完全なアーキテクチャを確認してください。

### 🌏 CJK 言語サポート（日本語、韓国語、繁体字/簡体字中国語）
- Jieba WASM（簡体字 + 繁体字中国語）、TinySegmenter（日本語）、Intl.Segmenter（英語/ラテン/韓国語）
- **言語ごとの専用ストップワードリスト** — 日本語、韓国語、繁体字中国語、簡体字中国語（および英語/ラテン）向けに、それぞれ別の辞書を用意しています。ストップワードとは、日本語の「の・は・を」、中国語の「的・地・得」、韓国語の「의・은・는・을」のように、どこにでも出てくるのに検索信号としてはほとんど意味を持たない文法的な埋め草です。キーワードスコアリングの前にこれらを取り除くことが、CJK コンテンツで BM25 のヒット率を高く保つ鍵になります——これをしないと、すべてのイベントがよくある助詞を含むだけでクエリに「マッチ」してしまい、本当に重要な単語が埋もれてしまいます。
- CJK トークナイザーモードは upsert 時に**各 Qdrant コレクションへロック**されます——モードを切り替えると警告モーダルが表示されます

### 🔍 ネイティブスパースベクトル・ハイブリッド検索（Qdrant）
上で説明した A3 パスです——グローバルに正確な BM25 IDF と、サーバーサイド RRF を使います。クエリごとの往復は 1 回だけです。

### 📝 保存前の要約
ベクトル保存前に、必ず LLM 要約を行います。OpenRouter とローカルの vLLM 互換エンドポイントに対応しています。プロンプトテンプレートも設定できます。

### ⏯️ より扱いやすいベクトル化コントロール
停止ボタン、一時停止/再開、そして Chrome が実行中に再起動しても残るフィンガープリントキャッシュを備えています。

### 🔒 チャットごとのコレクションスコープ
新しいコレクションは、現在のチャットに自動で有効化されます。「Active for current chat」チェックボックスでチャットロックを制御します。Database Browser のロックボタンでは、ロックが*この*チャット向けなのか、別のチャット向けなのかが分かります。

### 📡 より分かりやすいステータス表示
Auto-Sync カードには、現在のチャットがベクトル化済みかどうか、イベントがいくつあるかが表示されます。World Info カードには、ベクトル化済みのロアブックが名前付きで表示されます。不足しているものがあれば、どちらも正しいベクトル化ツールへリンクします。

### 🗂️ タブ式インターフェース
設定は **Core**（バックエンド、エンベディング、ハイブリッド）、**EventBase**（チャット履歴/アーカイブ .jsonl チャットファイル）、**ChunkBase**（ロアブック/ドキュメント/URL/wiki）、**Action**（診断、開発ツール）に分かれています。

### ⚡ 並列ウィンドウ — ベクトル化を高速化
長いチャットをベクトル化するとき、通常は 1 ウィンドウずつ処理します：ウィンドウ 1 を LLM に送る → 待つ → 埋め込む → ウィンドウ 2 を送る → 待つ → 埋め込む → … という流れです。2000 メッセージのチャットでは、これだけで大量の待ち時間が発生します。

**並列ウィンドウ**スライダー（Vectorize Content 内の Chunking Strategy セクション）を使うと、最大 **8 つの LLM 抽出 + 埋め込み呼び出し**を同時に走らせられます。ウィンドウ 1 が抽出中の間に、ウィンドウ 2–8 も同時に処理されるため、全体の取り込み時間を大きく短縮できます。

| スライダー値 | 動作 |
|---|---|
| **1（安全）** | 1 度に 1 ウィンドウ。プロバイダーへの負荷が最も低く、レートリミットのリスクもありませんが、最も遅いです。 |
| **2–4** | 軽めの並列処理。多くのプロバイダーでちょうど良いバランスです。 |
| **5–8（高速）** | 強めの並列処理。レートリミットが高いクラウドプロバイダー（OpenRouter、OpenAI、vLLM）に最適です。無料枠ではレートリミットに当たる可能性があります。 |

厳しいレートリミットの無料枠を使っている場合や、単一のローカル GPU で動かしている場合は **1** を使ってください。有料のクラウドプロバイダーを使っていて、2000 メッセージのチャット取り込みを 1 時間ではなく数分で終わらせたい場合は、**8** まで上げてください。

CJK の 1 文字フィルタリングを改善し、信号の強い 1 文字の RPG/日常生活/学校系語彙には、モード別の例外を用意しています。

混合バックエンド検索、handle ID フィルタリング、その他元の VectHare 由来の改善まわりで、多数のバグ修正を行っています。

## 🎭 アクティベーションルール

各コレクションカードにはアクティベーションパネルがあります。優先順位は次の通りです：

1. **無効**（一時停止ボタン）→ 決してクエリしない
2. **トリガー** → キーワードが最近のメッセージにマッチ → 有効化
3. **詳細条件** → トリガーが空、またはマッチしない場合に条件ルールを評価 → 有効化
4. **現在のチャットでアクティブ / キャラクターロック** → 手動で常時オンにするフォールバック
5. **何もマッチしない** → 有効化しない

条件では、感情（Character Expressions のスプライト検出経由）、キーワード、メッセージ数/ターン数、そして AND/OR を組み合わせたルールを使えます。

> ⚠️ **レガシー機能について：** トリガーと条件ベースのアクティベーションは**VectHare から継承した機能**で、後方互換性のためにのみ残されています。VectFox の大幅なアーキテクチャ変更により、**ユーザーは常にコレクションを「現在のチャットでアクティブ」にするか、キャラクターロックを使用すべき**で、トリガー/条件に依存すべきではありません。ベクトル検索の要点は、検索エンジンがすべてを検索できるようにすることです——キーワードトリガーに基づく選択的なアクティベーションは、その目的を損ないます。
> 
> **CJK 注意：** トリガーと感情/キーワード条件も**英語のみ**対応です——キーワード辞書が英語で、regex の `\b` 単語境界も CJK 文字の間では発火しません。中国語/日本語/韓国語のストーリーでは、代わりに**「現在のチャットでアクティブ」/ キャラクターロック**を使ってください。Message Count / Turn Count 条件は数値条件なので、どの言語でも問題なく動きます。

## 💾 インストール

### ステップ 1：拡張機能をインストール

1. ブラウザで SillyTavern を開きます
2. **Extensions** パネル（パズルピースのアイコン）へ移動します
3. **"Install Extension"** をクリックします
4. この URL を貼り付けます：
   ```
   https://github.com/KritBlade/VectFox
   ```
5. **Install** をクリックします

これで完了です！VectFox は自動でダウンロードされ、有効化されます。

### ステップ 2：Similharity プラグインをインストール（Qdrant は必須 — Standard は任意だが推奨）

**Qdrant バックエンドを使用する場合は必須です。** Standard バックエンドでは任意ですが、インストールすることで追加機能が利用できます：

| 機能 | Standard（プラグインなし） | Standard（プラグインあり） |
|---|---|---|
| イベント検索・注入 | ✅ 完全に動作 | ✅ 完全に動作 |
| エンベディング・ベクトル化 | ✅ 動作 | ✅ 動作 |
| リッチメタデータ（キーワード、重要度） | ❌ 保存されない | ✅ 保存・検索に活用 |
| データベースブラウザでチャンクを表示 | ❌ 利用不可 | ✅ 利用可能 |
| 個別チャンクの編集 | ❌ 利用不可 | ✅ 利用可能 |

Standard バックエンドでプラグインをインストールしなくても、イベント検索と注入は正常に動作します。チャンクの閲覧やキーワードメタデータは利用できなくなるだけです。

```bash
Windows ではコマンドプロンプト、Linux/Mac ではターミナルを開いてください。docker 上で使っている場合は Console に入ってください
cd SillyTavern/plugins
git clone -b Similharity-Plugin https://github.com/KritBlade/VectFox.git similharity
cd similharity
npm install
```

`config.yaml` で次のキーを探し、true に変更します：（Windows では SillyTavern\config.yaml、Linux/Mac では SillyTavern\config\config.yaml にあるはずです）
```yaml
enableServerPlugins: true
```
SillyTavern を再起動します。

### ステップ 3：VectFox を設定する
1. **VectFox Settings** を開きます（拡張機能パネルの Core タブ）。
2. ベクトルストレージを選びます（Standard または Qdrant）。
3. エンベディングプロバイダーを選びます (Transformers, vLLM, Ollama, OpenRouter, etc.).。
   - 💡 **おすすめ：** **OpenRouter** 経由で `qwen/qwen3-embedding-8b` を使ってください。非常に安価で、多言語対応に優れ（CJK・ラテン双方とも優秀）、VectFox が想定するコーパス規模に対して高品質な密ベクトルを生成します。
4. 要約 LLM を選びます（OpenRouter または vLLM）—— ベクトル化中の EventBase 抽出で使われます。
   - 💡 **OpenRouter のおすすめモデル：** 安価で高速な抽出パスには `openai/gpt-4o-mini` または `google/gemini-3.1-flash-lite` を使用してください——どちらもコストと取り込みのレイテンシを低く保てます。より高い抽出品質が必要で追加のコストを許容できる場合は、`x-ai/grok-4.3` がより強力ですが高価な選択肢です。OpenRouter が古いモデル ID（例：`x-ai/grok-4.1-fast`）に対して 404 や非推奨エラーを返す場合、その ID は避けてください。モデルの提供状況は時間とともに変わるため、長時間の取り込みを実行する前に OpenRouter 上で正確なモデル ID を確認してください。**Agent Mode の LLM**（AgentMode タブで別途設定）にも同じ推奨が当てはまります —— AgentMode のモデル欄を空欄のままにしておくと、この要約器設定が継承されます。
5. クラウドプロバイダー（OpenRouter / vLLM ）を使う場合は、API キーを設定します。
6. **Keyword Extraction** で、あなたのストーリーの言語を選びます。
7. ほとんどの設定はデフォルトのままで大丈夫ですが、必要に応じて自由に調整してください。
8. SillyTavern で対象のチャットを開き、もう一度 VectFox 拡張アイコンをクリックします。最初の DB をベクトル化するには、**必ず** "Vectorize Content" をクリックし、**Chat History** を選んでください。
9. 必要であれば、**AutoSync** タブで Auto-Sync を有効化します。頻度は EventBase タブの *Extraction > Window Size* で制御されます。
10. 必要であれば、**WorldInfo** タブでロアブック / World Info をベクトル化します。
11. （オプション）他の設定が動作することを確認したら、**AgentMode** タブで **Agent Mode** をオンにできます。provider / model / API key を空欄のままにすれば、要約器の設定が継承されます —— つまりステップ 4 で選んだ安価で高速なモデルがプランナーも動かしてくれます。詳しい仕組みは上の「どう動くのか → Agent Mode」セクションを参照してください。

## ❓ よくある質問

**なぜ内部に 2 つの別々のパイプラインがあるのですか？**
内部的には、VectFox はコンテンツの種類に応じて、次の 2 つの検索パスのどちらかへルーティングします：

| パイプライン | 処理するもの |
|---|---|
| **EventBase** | チャット履歴（ライブチャット + アップロードされた `.jsonl` アーカイブ） |
| **Standard（Chunk）** | その他すべて：ロアブック、キャラクターカード、URL、ドキュメント、wiki ページ、YouTube トランスクリプト |

この 2 つが互いのコンテンツを見ることはありません——そのため、同じチャットメッセージが 2 回検索される（イベントとして 1 回、生チャンクとして 1 回）ことはありません。普段は気にしなくて大丈夫です。単に、EventBase がチャットを担当し、標準チャンクパイプラインがそれ以外を担当する、という意味です。

**チャットの途中で CJK トークナイザーモードを変更できますか？**
できません——やらないでください。CJK トークナイザーモードは、upsert 時にセンチネルポイントを通して**各 Qdrant コレクションへロック**されます。すでにコンテンツをベクトル化したあとでモードを切り替えると、次のクエリ時に「Tokenizer mismatch」警告モーダルが表示されます。そのとき実際に選べるのは次のどちらかだけです：
1. 元のモードに**戻す**（既存のベクトルを保持）、または
2. 新しいモードで**コレクションを最初から再ベクトル化**する（抽出済みイベントをすべて捨ててやり直す）。

チャットをベクトル化*する前に*トークナイザーモードを選び、そのまま使い続けてください。インプレース移行はありません——スパースベクトルは元のモードでトークン化されているため、別のトークナイザー出力とは互換性がありません。チャットをベクトル化*する前に*エンベディングモデルも選び、そのまま使い続けてください。

**中国語/日本語/韓国語のチャットでトリガー/感情条件を使えますか？**
信頼できません。キーワード辞書は英語のみで、regex の `\b` 単語境界は CJK 文字の間では発火しません。「Active for current chat」/ キャラクターロック、または数値条件の Message Count / Turn Count を使ってください。

**なぜ「scene」設定が見つからないのですか？元の VectHare にはありました**
Scene サポートは削除されました（チャンクベースのチャット時代の機能で、現在のチャットは EventBase を通して処理されます）。イベントを無理にグループ化するのはあまり筋が良くないため、この機能は削除しました。

**システムの説明が技術的すぎて理解できません！**
同感です。この拡張機能は、一般的な SillyTavern チャットには過剰かもしれません。一体誰が 10,000 回以上も返信するチャットをするんですか？でも、検索品質は重要です。

**このエクステンションはマルチユーザーや公開環境での使用に対応していますか？**
VectFox + Similharity は**個人利用を前提に設計されています** — 自分のマシンまたは信頼できる閉域ネットワーク（LAN）での使用を想定しており、インターネット公開環境やマルチユーザー環境には適していません。主な理由が 2 つあります：

- **Qdrant** はデフォルトのインストールではユーザーごとのアクセス制御を持ちません。同一の Qdrant インスタンスを共有している場合、ポートに到達できる誰もが全コレクションの読み書きが可能です。2 人のユーザーが同じ Qdrant サーバーを共有すると、お互いのコレクションを読み書きできてしまいます。
- **Similharity プラグイン** にもユーザーごとの認証機能はありません。バックエンド（OpenAI、Ollama、vLLM など）へのエンベディング・クエリのリレーは、ユーザーレベルの分離なしで行われます。

Qdrant への RBAC 追加とプラグインへのユーザーごとの ACL 実装は、ドロップイン型のベクトルメモリ拡張のスコープを大きく超えるため、対応範囲外です。マルチユーザーの分離が必要な場合は、ユーザーごとに独立した Qdrant インスタンスと SillyTavern を実行するか、コンテナ化された環境を使用してください。

なお、SillyTavern 自体は `enableUserAccounts = true` によるユーザーごとのデータ分離をサポートしています。この機能は ST 自身のデータ（チャット、設定、キャラクター）をユーザーごとに分離しますが、共有 Qdrant サーバーには及びません。

## 🐛 トラブルシューティング

**"No embeddings available"** — Vectorize Content アイコンをクリックし、メイン ST 設定で Vectors 拡張機能を有効にし、エンベッディングプロバイダーを選択し、必要に応じて API キーを追加してから、Diagnostics を実行してください。

**イベント/チャンクが検索されない** — SillyTavernのロビーではなく、すでにチャット内にいることを確認してください。
Database Browser アイコンをクリック → チャットのコレクションをクリック → そのコレクションが "Active for current chat" になっていることを確認してください。

**「このメッセージで AI は実際に何を想起するのか？」** — Actions タブの **Debug Query** ボタンを使って「仮定」テストができます。テキストを入力するだけで、実際のチャットメッセージを送らずにライブデータベースに対してクエリを実行し、どのイベントが検索されるか、そのスコアも含めて正確に確認できます。閾値の調整や、重要なイベントが正しくインデックスされているかの確認に役立ちます。

**"Backend health check failed"** — Qdrant を使っている場合は、Qdrant サーバーが実行中で、Similharity プラグインがインストールされていることを確認してください。

**動作が遅い** — Qdrant + A3 に切り替えてください（1 回の往復、サーバーサイド融合）。EventBase Top K を減らしてください。ローカル GPU（順次処理）ではなく、API エンベディングプロバイダー（並列処理）を使ってください。

**大規模なチャット履歴のベクトル化が遅い** — VectFox はデフォルトで API レート制限を適用します（60 秒あたり 60 呼び出し）し、無料層バックエンドを保護します。ベクトル化を高速化するには：**設定 → Core タブ → Embedding → API Rate Limiting → \"Max Calls\" を 0 に設定して無効化**。本番 Qdrant の場合、サーバーの容量に応じて 100+ 呼び出し/分に増やすことができます。
