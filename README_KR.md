## 🎯 VectFox란?

뛰어난 VectHare 기반 위에 구축된 VectFox는 **SillyTavern을 위한 고성능 장기 기억 시스템**입니다. 단순한 다국어 포크를 훨씬 뛰어넘습니다 — 모든 지능형 검색 로직은 실제 벡터 데이터베이스(Qdrant) 서버 측에서 실행되며, 구조화된 이벤트 기반 접근 방식이 원시 청크 요약을 대체하여 극적으로 정확한 리콜을 실현합니다. 2,000개 이상의 메시지에서도 3초 이내에 결과를 반환합니다. 모든 언어 지원. 영어·일본어·한국어·번체/간체 중국어는 불용어 정밀 조정 포함.

원본 VectHare를 분기한 이유는 제 개인적인 [MVU Game Maker](https://github.com/KritBlade/MVU_Game_Maker) 프로젝트의 방대한 규모를 처리하기 위해서였습니다. 이 프로젝트들의 특징은:
- **극단적 규모: 스토리당 2,000개 이상의 응답, 응답당 1,000자 이상. 요약 검색은 3초 미만에 반환됩니다**
- 모든 언어 지원. 영어·일본어·한국어·번체/간체 중국어는 불용어 목록으로 추가 정밀 조정. 스페인어·독일어·아랍어·힌디어 등도 별도 설정 없이 동작합니다.
- [MVU Game Maker](https://github.com/KritBlade/MVU_Game_Maker)의 모든 기능 태그 제거.
- 실제로 작동하는 초장기 기억(2000+ 메시지)

일반적인 SillyTavern 메모리 익스텐션은 이런 부하에서 완전히 무너집니다. 특히 [MVU Game Maker](https://github.com/KritBlade/MVU_Game_Maker)가 사용하는 기능 태그가 스토리 안에 많이 존재할 때 더 그렇습니다. 그런 태그는 기억 조회에는 쓸모가 없거든요. 그래서 이런 기능 태그를 모두 정리하면서도 초대규모에서 고속 벡터화를 유지할 수 있는 도구가 필요했습니다.

대부분의 메모리 익스텐션은 메시지 100개 이하의 채팅을 위해 설계되었고, 그 규모에서는 완벽하게 동작합니다. 하지만 채팅이 그것을 넘어 커지면, 점점 더 공격적으로 오래된 메시지를 요약할 수밖에 없게 됩니다. 결국 최근 기록은 풍부한 디테일을 갖지만 오래된 것은 심하게 압축된 흐릿함이 되어버리고—사실 100+ 메시지 분량의 원시 컨텍스트를 프롬프트나 자동 생성된 로어북 항목에 도저히 욱여넣을 수 없기 때문에 진짜 해결책이 없습니다. 오래된 기억은 **반드시** 압축되어야 하고, 그것은 곧 디테일을 잃는다는 뜻입니다.

네모난 바퀴를 가진 차는 아무리 튜닝해도 이 문제를 해결할 수 없습니다. 그 일에 맞는 올바른 도구가 필요합니다. 제가 진짜로 필요로 하는 것은 이 모든 기억을 제대로 저장할 전용 벡터 데이터베이스 백엔드입니다.

***저는 SillyTavern을 위한 아키텍처적으로 올바르고 프로덕션에 가까운 설계의 메모리 시스템을 구축하기로 결정했습니다. 메모리를 하드코어하게 만들어봅시다!***

이 문제를 해결하기 위해 VectFox는 채팅의 **의미 있는 모든 이벤트**를 저장하는 전용 벡터 데이터베이스를 사용합니다. 첫 번째 메시지든 2,000번째 메시지든, 의미 있는 모든 이벤트는 데이터베이스에 남아있고 항상 SillyTavern이 검색할 수 있습니다. 제가 원하는 것은 10k+ 메시지로 확장 가능하고 라운드트립 검색이 수 초 이내에 끝나는 SillyTavern용 프로덕션급 메모리 벡터 시스템입니다.

### 해결하는 문제

- 🧠 **원본 VectHare는 무엇을 기억해야 하는지 생각하지 않습니다.** VectFox는 그 위에 LLM 기반 **EventBase** 추출 레이어를 추가합니다: AI가 어떤 순간이 의미 있는 이벤트인지 스스로 판단하고, 각 이벤트에 관련된 캐릭터, 아이템, 장소, 개념을 태그합니다.
- 🤖 **원본 VectHare는 질문을 추론하지 않습니다** — 표면적인 텍스트 유사도만 매칭할 뿐입니다. VectFox는 선택적인 **Agent Mode**를 추가하여, 작은 LLM이 다각도 검색을 계획해 사용자의 원본 쿼리만으로는 찾을 수 없었던 기억까지 끌어올립니다.
- 😩 메모리 저장 전에 [MVU Game Maker](https://github.com/KritBlade/MVU_Game_Maker)가 사용하는 모든 기능 태그를 제거합니다.
- 🧠 [MVU Game Maker](https://github.com/KritBlade/MVU_Game_Maker)의 캐릭터 기반 기억 위에 스토리 기반 기억을 추가합니다.
- 💸 긴 대화가 무관한 이력으로 토큰 예산을 막아버리는 문제를 줄여줍니다.
- ✍️ 캐릭터에게 핵심 이벤트를 상기시키기 위해 컨텍스트를 수동으로 편집할 필요가 없어집니다.

**VectFox의 해결책:** 채팅이 얼마나 길어지든 채팅의 모든 의미 있는 이벤트를 저장하는 전용 벡터 데이터베이스로 **Qdrant**를 사용. 추가 서비스를 실행할 준비가 되지 않은 사용자를 위해, **A1과 A2 경로를 사용하는 "라이트 버전"** 이 SillyTavern 내장 벡터 스토어 위에서 추가 소프트웨어 없이 실행됩니다—풀 벡터 DB의 많은 기능을 소규모로 공유합니다. 진정한 장기 기억 시스템이 필요해지면 Qdrant를 사용하는 **A3 경로**로 업그레이드하세요.

참고: Qdrant는 무료 오픈 소스입니다.

### VectFox가 해결하려 하지 않는 것

VectFox는 **기억** 시스템이지, 트래커가 아닙니다. 퀘스트 진행, 캐릭터 스탯, 실시간 월드 상태를 추적하지 않습니다. 그런 기능이 필요하다면 [MVU Game Maker](https://github.com/KritBlade/MVU_Game_Maker)와 함께 사용하세요 — 퀘스트, 캐릭터, 스탯을 위한 GUI가 내장된 캐릭터 기반 트래킹 시스템입니다. 두 시스템을 함께 사용하면 긴 SillyTavern 롤플레이에서 발생하는 기억 및 상태 관련 문제의 약 90%를 해결할 수 있습니다.

## 🧠 작동 방식

벡터 검색은 정말 똑똑한 "찾기" 기능 같은 것입니다. 정확한 단어를 일치시키는 대신 **의미**를 일치시킵니다—"배고파"를 입력하면 "점심 먹으러 가자"라고 한 메시지를 찾을 수 있는데, *의미*가 비슷하기 때문이죠.

VectFox는 함께 작동하는 두 가지 아이디어를 중심으로 구축되었습니다:

### 1. EventBase — 응답별 요약이 아닌, 윈도우에서 추출된 이벤트

대부분의 메모리 익스텐션은 각 (또는 여러) AI 응답을 하나의 텍스트 덩어리로 요약합니다. 그럴듯하게 들리지만, 응답에 실제로 무엇이 들어 있는지 살펴보면:

- 100문장짜리 응답은 95문장의 채움 대사, 장면 설정, 잡담에 묻혀 있는 **5개의 의미 있는 이벤트**(전투, 발견, 약속, 아이템 교환, 관계 전환점)를 포함할 수 있습니다
- 다른 응답은 **0개의 이벤트**—그냥 잡담일 수도 있습니다
- 또 다른 응답은 1000자에 **1개의 이벤트**를 담을 수도 있죠.

응답별 요약은 이 세 가지 경우를 모두 "응답당 한 덩어리"로 평면화합니다—이벤트 경계를 잃고, 중요한 박자와 채움을 섞고, 실제로 무엇이 일어났든 똑같은 형태의 데이터를 생성합니다.

**EventBase는 최근 메시지의 윈도우를 보고 실제로 일어난 것에 따라 0, 1, 또는 여러 개의 구조화된 이벤트를 추출합니다.** 각 이벤트는 풍부한 메타데이터(`characters`, `locations`, `items`, `concepts`, `importance`, `DateTime` 등)를 가진 자체 레코드입니다—단순한 설명 문자열이 아니라.

Tav가 다른 팀원 3명과 배경 스토리 노이즈 사이에서 100문장에 걸친 긴 쇼핑 여행을 Astarion과 가졌다면, EventBase는 그 텍스트 더미에서 하나의 이벤트를 추출할 수 있습니다:

```
{
event_type:    item_acquired
importance:    6
text:          타브와 아스타리온이 발더스 게이트에서 갑옷을 쇼핑했다. 아스타리온은 가격을 비웃었다.
               타브는 가죽 흉갑을 80gp에 구매했다.
DateTime:      1492-08-15T14:00:00
cause:         타브는 샤르의 시련 원정 전에 더 나은 갑옷이 필요했다
result:        타브는 이제 가죽 흉갑을 착용하고 있다. 파티 자금에서 80gp가 사용됐다
characters:    [타브, 아스타리온]
locations:     [발더스 게이트, 소서러스 선드리스 지구]
factions:      []
items:         [가죽 흉갑, 80gp]
concepts:      [갑옷 쇼핑, 파티 경제]
keywords:      [갑옷, 가죽, 흉갑, 골드, 쇼핑]
open_threads:  [샤르의 시련 준비]

}
```

나중에 "그 쇼핑 여행 기억나?"라고 언급하면 VectFox는 **그 이벤트**를 검색합니다—100개의 원시 문장이 아니고, 쇼핑 여행을 둘러싼 무관한 잡담과 함께 평균화한 흐릿한 요약도 아닙니다.

### 2. 전용 벡터 DB가 왜 자연스러운 선택인가

응답별 요약만 한다면 사실 벡터 데이터베이스가 *필요하지 않습니다*. 응답당 ~1개 덩어리, 항상 같은 모양으로 생성하고, 채팅이 커짐에 따라 재귀적으로 다시 요약합니다. 간단한 텍스트 파일이면 충분합니다—그래서 많은 오래된 메모리 익스텐션이 진짜 DB를 굳이 쓰지 않은 거죠.

하지만 일단 EventBase에 전념하기 시작하면 상황이 달라집니다:

- 2,000개 응답 → 잠재적으로 **1,000–3,000개의 구조화된 이벤트** (몇몇 응답은 여러 개를 추출하고, 어떤 응답은 0개)
- 각 이벤트는 풍부한 필드를 가집니다: characters, locations, items, concepts, keywords, importance, timestamps
- 사용자가 대화 중일 때 실시간으로 **의미 + 키워드 + 메타데이터 필터링**으로 관련 있는 5–10개 이벤트를 찾아야 합니다

이것이 바로 **Qdrant 같은 전용 벡터 데이터베이스가 설계된 워크로드**입니다: 밀집 벡터 유사도와 희소 키워드 검색 모두에 메타데이터 필터링까지 더하고, 전체 코퍼스에 걸친 글로벌 BM25 가중치까지 갖춘 구조화된 작은 레코드들이 많은 경우. 평평한 요약 파일로 이걸 하려고 하면 선형 스캔, 키워드 인덱싱 없음, 메타데이터 필터 없음, 수백 개 항목 이상으로 확장 불가능을 의미합니다.

EventBase는 Qdrant를 *강제*하지 않습니다—A1/A2 라이트 경로는 SillyTavern 내장 Vectra에서 실행됩니다. 하지만 채팅이 수백 이벤트를 넘으면 Qdrant가 이런 데이터 모양을 위해 실제로 만들어진 저장 레이어입니다.

### 3. Agent Mode — AI가 검색을 계획하게 두기 (선택사항, Qdrant / A3 전용)

순수 벡터 검색에는 한 가지 약점이 있습니다: 당신이 문자 그대로 입력한 것만 검색할 수 있다는 점입니다. *"마이라, 내가 왜 너의 몸값을 지불했는지 기억해?"* 라고 물으면 검색은 그 단어들과 일치하는 이벤트를 찾습니다. 하지만 진짜 답은 몸값으로 이어진 납치, 가격이 결정된 협상, 그리고 그 후 당신 캐릭터의 감정적 반응에 걸쳐 있을 수도 있는데 — 당신의 원본 질문에는 그중 어느 것도 언급되어 있지 않습니다.

**Agent Mode는 작은 "플래너" LLM을 추가합니다**. 이 모델은 최근 대화와 의미적으로 가장 가까운 매치들을 읽고 묻습니다: *"이 질문에 제대로 답하려면 또 어떤 각도에서 검색해야 할까?"* 1–4개의 후속 검색 쿼리를 출력하고 Qdrant에 병렬로 보냅니다.

**구체적 예시.** 당신이 입력합니다:
> *마이라에게 말한다: "내가 왜 너의 몸값을 지불했는지 기억해?"*

Agent Mode가 없으면 검색이 찾는 것:
- ✓ 몸값 지불 이벤트 자체 ("몸값" 직접 매칭)

Agent Mode를 켜면 플래너가 자동으로 이런 각도들을 추가합니다 (이 종류의 질문에 대한 플래너 출력 예시):

```json
{
  "queries": [
    "마이라 몸값 지불 이유",
    "마이라 과거 버려질 두려움",
    "Critblade 아버지 찾는 약속",
    "마이라 리더십 시험 순간"
  ],
  "filters": {
    "characters_any": ["Critblade", "마이라"],
    "concepts_any": ["몸값", "버림", "리더십"]
  }
}
```

네 개의 병렬 Qdrant 검색이 반환하는 것:
- ✓ 몸값 지불 (직접)
- ✓ 마이라가 과거에 버려질까봐 두려워했던 순간 (감정적 배경)
- ✓ 사라진 아버지를 찾아주겠다는 약속 (플래너가 최근 대화 맥락에서 추론한 관련 서사 비트)
- ✓ 리더십이 시험받던 순간 (플래너가 가져온 관련 아크)

네 결과는 원본 검색 결과와 병합되어 같은 4가중치 재랭커를 통과합니다. 메인 응답 LLM은 그 순간의 지불뿐만 아니라 **완전한 인과 체인과 감정적 배경**까지 받게 됩니다.

**Agent Mode가 A3 (Qdrant)와 짝을 이루는 이유**:
- 각 플래너 쿼리는 독립된 Qdrant 호출입니다. Qdrant는 4개의 병렬 쿼리를 1–3초 만에 처리합니다.
- AgentMode는 Qdrant가 필요합니다 — 표준 백엔드에서는 완전히 건너뜁니다 (`mode=SKIPPED reason=requires_qdrant_backend` 로그) 그리고 사전 검색을 그대로 반환합니다. 우아한 폴백 없음, 의도된 설계.
- Qdrant의 페이로드 필터(`characters_any`, `locations_any`, `factions_any`, `items_any`, `concepts_any`, `event_type_any`, `importance_gte`)는 플래너가 각 검색을 정확히 좁힐 수 있게 해줍니다 — 표준 백엔드는 이를 노출하지 않습니다.

**비용 및 지연**: 플래너로 GPT-4o-mini 또는 Haiku 같은 작은 모델을 사용하면 턴당 약 $0.0002, 추가 지연은 약 2–5초. **순수하게 가산적**입니다 — 일반 검색을 대체하지 않으며, 플래너가 실패하면 표준 흐름으로 깔끔히 폴백합니다. 전용 **AgentMode** 탭에서 설정합니다. 기본값은 꺼짐.

> 💡 Agent Mode는 **반성적 질문**("왜...", "...을(를) 기억해?"), **모호한 회상**("아버지에 관한 그 일"), 그리고 **인과 체인 쿼리**처럼 — 문자 그대로의 메시지가 검색 앵커를 포함하지 않는 경우에 가장 잘 작동합니다. 메시지에 이미 정확한 키워드가 들어있는 직접적 조회라면 표준 사전 검색만으로도 대부분의 일이 해결됩니다. 그런 경우 Agent Mode는 부가적인 마감 다듬기 역할입니다.

### 🧠 이것이 기존 메모리 익스텐션보다 나은 이유

대부분의 기존 메모리 익스텐션은 두 가지 접근 방식 중 하나를 사용합니다. 둘 다 채팅이 커짐에 따라 디테일을 잃습니다. 이유는—그리고 EventBase가 이를 어떻게 피하는지:

| 측면 | 📝 롤링 요약 <br>*(대부분의 "메모리" 익스텐션)* | ✂️ 원시 청크 분할 <br>*(오래된 벡터 RAG)* | 🧬 EventBase <br>*(VectFox)* |
|---|---|---|---|
| **저장되는 것** | 끝없이 자라는 하나의 요약 텍스트 | 원시 청크로 잘린 모든 메시지 | 메타데이터를 가진 구조화된 이벤트 레코드 |
| **메시지 100 시점** | 대부분 그대로 | 그대로 | 그대로 |
| **메시지 200 시점** | 심하게 압축됨—이름, 숫자, 일회성 디테일이 표류하거나 사라짐 | 토큰 예산 오버플로우—오래된 청크는 점수 기준 가지치기 또는 삭제 | **그대로**—오래된 이벤트는 여전히 DB에 있고, 관련성으로 표면화 |
| **메시지 1,000+ 시점** | 사실상 흐릿함 | DB 부풀어 오름; 원시 청크가 저신호라 검색이 시끄러워짐 | **그대로**—현재 장면에 관련된 소수의 이벤트만 추출됨 |
| **"압축"이 하는 일** | 요약을 재귀적으로 재요약, 매번 정보 손실 | 없음—그러나 합성도 없음; 원시 텍스트는 검색에 적중 또는 미스 | 1회성, 의미적—*의미 있는 이벤트*를 추출하고 채움을 버림. 이벤트 자체의 디테일은 보존됨. |
| **검색 신호** | 없음—요약 전체가 항상 주입됨 | 원시 텍스트에 대한 벡터 유사도 (의역을 잡지만 노이즈도 많음) | 풍부한 필드(`characters`, `items`, `locations`, `concepts`, `keywords` + 밀집 의미)에 대한 벡터 + BM25 하이브리드 |
| **디테일은 어디로 가나** | 압축되면 영원히 손실 | 청크가 점수 임계값 아래로 떨어지면 손실 | **아무 데도 안 감**—이벤트는 벡터 DB에 살아있고 관련 시 표면화 |
| **무엇이 주입되나** | 실행 중인 요약 전체 (매 턴, 매번) | 의미적으로 가까운 몇몇 원시 메시지 | 현재 메시지에 중요한 이벤트만 |

**핵심 통찰:** 롤링 요약이 디테일을 잃는 것은 공간을 확보하기 위해 오래된 내용을 *버리기* 때문입니다. 원시 청크 분할이 디테일을 잃는 것은 *검색이 규모에서 무너지기* 때문입니다. EventBase는 의미 있는 모든 이벤트를 영원히 보존하면서—벡터 + 키워드 검색이 지금 AI에게 보여줄 가치가 있는 5–10개를 결정하도록 합니다. 디테일은 압축되지 않습니다; **무관함이 필터됩니다**.

> 💡 **메시지를 어떻게 표현하는지가 무엇이 검색되는지에 큰 영향을 줍니다.** 검색은 당신 응답의 텍스트에 의해 구동되므로, 사용하는 단어가 중요합니다. 예를 들어, *"Mayla, 내가 왜 몸값을 지불했는지 기억해?"* 와 *"Mayla, 내가 왜 2000달러를 냈는지 기억해?"* 는 매우 다른 이벤트를 반환합니다—"몸값"은 그 스토리라인과 연결된 모든 이벤트(납치, 협상, 전달)를 끌어오지만 "2000달러"는 주로 문자 그대로 2000이라는 숫자를 언급하는 이벤트와만 일치합니다. AI가 특정 장면을 회상하길 원한다면, 정확한 숫자 같은 부수적인 디테일이 아닌, 그 장면의 **스토리적으로 의미 있는 단어**로 메시지를 정착시키세요.
>
> 1,500개 이벤트 규모의 채팅에서 나란히 비교 테스트한 결과, A3(Qdrant)는 적절히 정착된 쿼리에서 몸값 이벤트를 **#1 / #2**로 랭크했고, 숫자 중심 쿼리에서도 여전히 상위에 표면화했습니다. A1 / A2(표준 백엔드)도 **같은 몸값 이벤트를 찾기는 했지만** 더 낮게 랭크했습니다(적절히 정착된 쿼리에서도 약 **#3 / #5**). 이는 실제로 프롬프트에 주입되는 상위 이벤트 범위를 벗어나는 경우가 많은 순위입니다. 차이는 구조적입니다 — A3는 희소 키워드 인덱스를 통해 전체 코퍼스를 검색하지만, A1 / A2는 밀집 벡터 레이어가 우연히 가져온 후보만 점수화합니다(아래 경로 비교 참조). 정착어는 모든 백엔드에서 중요하지만, A3는 잘못 추측했을 때 더 관대합니다.

## 🔍 하이브리드 검색: A1 vs A2 vs A3 경로

VectFox는 최상의 결과를 찾기 위해 **두 가지 신호**를 결합합니다:

- **신호 1 - 벡터 유사도** — 의미 기반 ("배고파"가 "점심 먹으러 가자"와 일치)
- **신호 2 - BM25 키워드 점수** — 정확한 단어 일치 ("Astarion"이 "Astarion"과 일치)

백엔드와 설정에 따라 이를 결합하는 **세 가지 경로**가 있습니다: (브라우저에서 docker의 전용 벡터 데이터베이스까지)

### A1 — Standard 백엔드 + BM25
VectFox는 먼저 "의미"로 당신의 메시지와 *관련 있어 보이는* 이벤트를 **약 100개** 찾고, 그 100개에 대해서만 당신이 쓴 정확한 단어를 다시 확인한 뒤 두 점수를 합칩니다. 가볍고 빨라서 느린 컴퓨터에 잘 맞습니다.

**다만 주의:** 완벽한 키워드를 가진 이벤트가 처음의 그 100개에 들지 못했다면, 그 이벤트는 한 번도 검토되지 않습니다 — 키워드 단계에서는 보이지 않는 존재가 됩니다.

<details>
<summary>⚙️ 기술 세부사항</summary>

브라우저가 벡터 검색을 수행해 상위 ~100개 후보를 얻고, 그 후보에 대해서만 BM25 키워드 점수를 계산합니다. 단순한 가중합: `α × vectorScore + β × bm25Score`.

</details>

### A2 — Standard 백엔드 + Hybrid (A3 경로를 가지 않는 대부분의 사용자에게 권장)
첫 단계는 A1과 같습니다 — *관련 있어 보이는* 이벤트 약 100개를 가져옵니다 — 하지만 A2는 **순서를 더 똑똑하게** 매깁니다. 같은 100개를 두 가지 방식(의미로 한 번, 키워드로 한 번)으로 랭크하고 두 목록을 융합한 뒤, *두* 신호 모두에서 좋은 점수를 받은 이벤트에 보너스를 줍니다. A3 경로를 가지 않는 대부분의 사용자에게 권장합니다.

**다만 주의:** 문제는 A1과 같습니다. A2는 이미 가져온 100개를 재정렬할 수 있을 뿐, 첫 단계에서 놓친 키워드 일치를 나중에 끌어올 수는 없습니다. (이를 해결하는 것은 A3뿐입니다.)

<details>
<summary>⚙️ 기술 세부사항</summary>

브라우저가 벡터 검색을 수행해 상위 ~100개 후보를 얻은 뒤, **동일한 후보 풀**을 두 가지 방식으로 랭크합니다 — 벡터 유사도로 한 번, BM25로 한 번 — 그리고 두 랭크된 목록을 다음 방식으로 융합합니다:
- **RRF (Reciprocal Rank Fusion, 역순위 융합)** — 원시 점수가 아닌 각 목록 내 *위치*로 결과를 결합
- **이중 신호 보너스** — *두* 목록 모두에 나타나는 결과는 최대 +8% 부스트, 단일 신호 결과는 작은 페널티 (벡터만 ×0.55 / 텍스트만 ×0.60)

> ⚠️ **A2는 독립적인 희소 검색이 아닙니다.** 두 랭크 목록 모두 *동일한 밀집 ANN 후보 풀*에서 가져옵니다. BM25는 밀집 레이어가 이미 반환한 후보를 재정렬할 수는 있지만, 밀집 ANN이 놓친 키워드 전용 일치를 표면화할 수는 없습니다. 이벤트마다 진짜 희소 벡터를 저장하고 전체 코퍼스에 대해 희소 검색을 실행하는 것은 **A3**뿐입니다.

**예:** "Astarion drinks blood." 검색. 밀집 ANN이 Astarion과 피를 언급하는 이벤트를 반환했다면 (예: "뱀파이어/배고픔"이 의미적으로 가깝기 때문에) 두 랭킹 모두 그것을 표면화하고 이중 신호 보너스가 더 끌어올립니다. 만약 희귀 키워드 전용 이벤트가 상위 100 윈도우 밖에 있다면, A2는 찾지 못합니다 — A3는 찾을 것입니다.

**트레이드오프(기술):** 더 빠른 컴퓨터 브라우저에서 더 나은 융합을 제공하지만, 리콜은 여전히 밀집 top-K(~100 후보)로 제한됩니다.

</details>

### A3 — Qdrant 네이티브 스파스 + 서버측 RRF + 포뮬라 재랭크 (최고의 정확도)
A3는 진짜 큰 업그레이드입니다. 먼저 ~100개를 가져오는 대신, Qdrant가 **당신의 전체 기록을 두 가지 방식으로 동시에 검색**합니다 — 의미로, 그리고 정확한 키워드로 — 그리고 모든 랭킹과 필터링을 서버측에서 단 한 번의 요청으로 처리합니다. 그래서 1,500개 채팅 전의 어떤 이벤트에 묻혀 있는 희귀한 단어라도, 첫 선발에 들지 못해 놓치는 일 없이 곧바로 찾아냅니다.

**AgentMode**를 쓰면 한 걸음 더 나아갑니다: 당신의 질문을 여러 각도(*이 일이 어떻게 일어났나? 그 전에 무슨 일이 있었나? 누가 또 관여했나?*)로 분해하고, 검색하기 *전에* 누구·어디·무엇으로 범위를 좁혀서 관련 없는 이벤트가 자리를 차지하지 않도록 합니다.

**트레이드오프:** 최고의 정확도, 규모에서 가장 빠름. Qdrant 인스턴스가 필요합니다(무료, 오픈 소스). → [Qdrant 설치 가이드](Doc/Qdrant_install.md)

<details>
<summary>⚙️ 기술 세부사항</summary>

A3는 **모든 것을 Qdrant 내부의 단일 API 호출에서** 실행합니다. A1 / A2 대비 진짜 구조적 강점은 단순히 "더 빠르다"가 아닙니다 — **A3는 키워드로도 전체 코퍼스를 실제로 검색**하는 반면, A1 / A2는 밀집 벡터로만 검색한다는 점입니다:

1. **전체 코퍼스에 대한 밀집 + 희소 하이브리드 검색** — Qdrant는 upsert 시점에 이벤트마다 희소 키워드 벡터를 저장합니다. 쿼리 시점에 밀집 인덱스(의미)와 희소 인덱스(키워드)가 **컬렉션의 모든 이벤트에 대해 병렬로 실행**되고, 두 결과 목록이 네이티브 RRF로 융합됩니다. 이것이 A3가 진정으로 확장되는 이유입니다: 희귀한 키워드가 1,500개 채팅 전의 한 외진 이벤트에만 등장하더라도, 희소 인덱스는 그것을 직접 찾아냅니다 — 밀집 벡터 레이어가 우연히 그것을 표면화하는지 여부와 무관합니다. (A1 / A2는 밀집 ANN이 반환한 top-K 후보만 점수화하므로, 그 창 밖의 이벤트는 키워드가 아무리 잘 일치해도 보이지 않습니다.) BM25 IDF도 서버측에서 전역적으로 계산되므로 희귀어의 점수가 구조적으로 정확합니다.

2. **서버측 포뮬라 재랭크** — Qdrant가 동일한 호출 내에서 그 하이브리드 결과에 4-가중치 재랭크 공식을 적용합니다. 각 `w_X`는 사용자 조정 가능한 가중치 슬라이더이고, 곱해지는 항은 가중치가 깔끔하게 합성되도록 정규화됩니다: `w_cosine × RRF_score + w_importance × (importance/10) + w_persist × (1 if should_persist else 0) + w_recency × exp_decay(source_window_end → chatLength)`. 최종 랭크된 목록이 이미 정렬된 상태로 반환됩니다. 추가 왕복도 없고 브라우저 JavaScript의 점수화도 없습니다.

3. **서버측 필터링** — 최소 중요도 임계값과 컨텍스트 중복 제거 컷오프(최근 채팅에 이미 보이는 이벤트)가 결과가 도착한 후가 아니라 Qdrant 내부에서 적용됩니다. 임계값 아래의 이벤트는 서버를 떠나지 않습니다.

4. **AgentMode 시맨틱 사전 필터링 + 다각도 쿼리** — AgentMode는 서로를 강화하는 두 가지 작업을 수행합니다. 첫째, 플래너 LLM이 사용자의 질문을 여러 하위 쿼리로 분해합니다 — 표면적 의미뿐만 아니라 *이 일이 어떻게 일어났나?*, *그 전에 무슨 일이 있었나?*, *결과는 무엇이었나?*, *누가 관여했나?* 같은 각도도 포함하며, 각 각도는 별도의 벡터 검색으로 실행됩니다. 둘째, 플래너가 구조화된 엔티티 필터(`characters_any`, `locations_any`, `factions_any`, `items_any`, `concepts_any`, `event_type_any`, `importance_gte`)를 발행하며, 이는 동일한 호출 내에서 Qdrant 페이로드 절로 적용되어 벡터 검색이 실행되기 *전에* 후보 풀을 좁힙니다. 두 기능의 조합은 강력합니다: 다각도 쿼리는 광범위한 시맨틱 그물을 던지고, 핀포인트 필터는 각 검색이 올바른 엔티티로 범위가 한정되도록 보장합니다 — 관련 없는 캐릭터, 위치, 이벤트 유형은 리콜 슬롯을 차지하지 않습니다.

브라우저는 앵커 부스트(구문 매칭), 페어와이즈 중복 제거, 그리고 여러 컬렉션에 걸친 최종 병합만 처리합니다.

**예:** "I cast Fireball at the dragon." 검색. Qdrant가 밀집(주문/공격 의미)과 희소(문자 그대로 "Fireball" + "dragon")를 동시에 검색하고, RRF로 융합하고, importance/recency 포뮬라로 순위를 매기고, 낮은 중요도 이벤트를 필터링한 다음—최종 주입 준비된 목록을 한 번의 호출로 반환합니다.

**AgentMode 예:** "Shadowheart와 무슨 거래를 했나요?" 플래너가 `characters_any: ["Shadowheart"]`를 발행하고 하위 쿼리로 확장합니다: 원래 질문, *"Shadowheart와 관련된 합의나 약속"*, *"Shadowheart와의 거래로 이어진 사건"*, *"Shadowheart가 대가로 요구한 것"*. 각 하위 쿼리는 이미 Shadowheart 태그 이벤트로 제한된 Qdrant 후보 풀에 대해 실행됩니다 — 광범위한 시맨틱 리콜, 크로스 캐릭터 노이즈 없음.

</details>

## ✨ 기능

### 🧬 EventBase — LLM이 추출한 채팅 기억
LLM이 채팅을 중요도/최근성/지속성 가중치를 가진 구조화된 이벤트로 요약합니다. 4-가중치 재랭커가 무엇이 주입될지 결정합니다. 내장된 중복 제거는 최근 메시지에 이미 보이는 이벤트를 억제합니다.

각 이벤트는 원시 텍스트가 아닌 구조화된 레코드입니다. LLM이 Astarion 쇼핑 윈도우에서 생성하는 것의 예:

```
{
event_type:    item_acquired
importance:    6
text:          타브와 아스타리온이 발더스 게이트에서 갑옷을 쇼핑했다. 아스타리온은 가격을 비웃었다.
               타브는 가죽 흉갑을 80gp에 구매했다.
DateTime:      1492-08-15T14:00:00
cause:         타브는 샤르의 시련 원정 전에 더 나은 갑옷이 필요했다
result:        타브는 이제 가죽 흉갑을 착용하고 있다. 파티 자금에서 80gp가 사용됐다
characters:    [타브, 아스타리온]
locations:     [발더스 게이트, 소서러스 선드리스 지구]
factions:      []
items:         [가죽 흉갑, 80gp]
concepts:      [갑옷 쇼핑, 파티 경제]
keywords:      [갑옷, 가죽, 흉갑, 골드, 쇼핑]
open_threads:  [샤르의 시련 준비]

}
```

두 신호(의미 + 키워드)가 이 풍부한 필드 세트에 대해 작동하므로 "armor for the dungeon" 같은 쿼리는 concepts/open_threads를 통해 적중하고 "Astarion 80gp"는 characters/items/keywords를 통해 적중합니다. 이 구조는 Qdrant 벡터 데이터베이스에 네이티브이므로 적중률이 다른 어떤 메모리 익스텐션보다 훨씬 높습니다.

### 🤖 Agent Mode — LLM이 계획하는 다각도 검색 (Qdrant / A3 전용)
작은 플래너 LLM이 최근 대화와 사전 검색의 상위 후보를 읽고, 사용자의 질문의 서로 다른 각도를 노린 1–4개의 후속 Qdrant 쿼리를 출력합니다. 결과는 병렬로 가져와 같은 4가중치 재랭커를 통해 병합됩니다 — **순수하게 가산적이며, 일반 검색을 대체하지 않습니다**. 어떤 실패에서도 표준 흐름으로 깔끔히 폴백합니다. 문자 그대로의 메시지에 검색 앵커가 포함되지 않는 **반성적 질문, 모호한 회상, 인과 체인 쿼리**에서 가장 효과적입니다.

- **프로바이더/모델 상속** — AgentMode 탭에서 비워두면 요약기 설정을 상속합니다. 더 저렴한 모델(예: Haiku 4.5, GPT-4o-mini)로 오버라이드하여 플래너 비용을 줄일 수 있습니다.
- **언어 매칭 프롬프트** — 플래너는 대화 언어(중국어, 일본어, 한국어, 라틴 문자, 영어)로 쿼리를 출력하고 고유명사는 원형을 유지합니다. 언어 간 오염이 없습니다.
- **설정 가능한 슬라이더** — 플래너에 보낼 과거 턴 수(1–10), 플래너에 보여줄 후보 수(5–20), 최대 쿼리 수(1–4), 타임아웃(1–60초), 디버그 로깅.
- **실제 비용** — 작고 빠른 모델 기준 턴당 약 $0.0002, 추가 지연 2–5초. 기본값 꺼짐 — 장편 회상이 필요할 때 켜십시오.

자세한 아키텍처는 설정의 **AgentMode** 탭, 또는 위의 "작동 방식 → Agent Mode" 섹션을 참조하세요.

### 🌏 다국어 지원

VectFox는 모든 언어를 토크나이즈하고 인덱싱할 수 있습니다——BM25/키워드 처리는 완전히 언어 중립적입니다. 모든 언어는 별도 설정 없이 바로 동작하며, 5개 언어는 불용어 튜닝이 추가로 적용됩니다.

|---|---|---|---|
| **영어** | Intl.Segmenter | ✅ 667개 | 항상 활성화되는 기준선. 모든 모드에 적용 |
| **일본어** | TinySegmenter | ✅ 672개 | 「は・を・の…」 등 조사 제거 |
| **한국어** | Intl.Segmenter | ✅ 679개 | 「의・은・는…」 등 조사 제거 |
| **번체 중국어** | Jieba WASM (TW 사전) | ✅ 899개 | `jieba_tw` 모드 전용. 간체와 별도 관리 |
| **간체 중국어** | Jieba WASM | ✅ 994개 | `jieba` 모드 전용. 번체와 별도 관리 |

**불용어 목록은 전체 합산이 아닌 모드별 선택 방식입니다.** 한국어 컬렉션은 영어+한국어(약 1,346개)만 참조합니다——이전처럼 2,200개 이상의 중국어 항목을 검색하지 않습니다. 이로써 한·중 공통 한자가 중국어 조사와 우연히 일치해 색인에서 무음으로 사라지는 문제가 해결됩니다.

**결합 문자 스크립트**(인도계: 힌디어·타밀어·벵골어 등, 아랍어 하라카트 등)는 단어 단위로 토크나이즈됩니다——마트라와 비라마(`्`)가 보존되어 `युद्ध`("전쟁")가 조각나지 않고 온전히 인덱싱됩니다.

**언어 불용어 목록 추가**는 파일 1개 PR로 완결됩니다: 단어 배열 추가 + 로케일 레지스트리에 1줄 + 모드 테이블에 레코드 1개. 그 외에는 아무것도 바꿀 필요가 없습니다.

- CJK 토크나이저 모드는 upsert 시 **각 Qdrant 컬렉션에 고정**됩니다—모드 전환은 경고 모달을 표시합니다

### 🛡️ 견고한 벡터화 — 무슨 일이 있어도 끝까지 완료

긴 스토리 × 불안정한 연결이야말로 순진한 벡터라이저가 죽는 지점입니다. 임베딩 호출이 한 번 멈추거나, 타임아웃이 한 번 발생하면 2,000개 메시지 인제스트가 중간에 무너집니다. VectFox는 벡터화가 **항상 전진하고 항상 복구되도록** 설계되었습니다 — 각기 다른 장애 모드를 봉쇄하는 3개의 독립적인 안전망이 있습니다.

- **요청 헤지(hedge)** — 임베딩 호출이 임계값(기본 15초)을 넘어 응답이 없으면, VectFox는 계속 기다리는 대신 새 연결로 *복제* 요청을 발사합니다. 최대 3개의 헤지가 원본과 경쟁하고, 가장 먼저 응답한 것이 이기며, 늦은 것들은 폐기됩니다. SiliconFlow 워커 하나가 막히거나 OpenRouter 라우팅 판단 하나가 잘못되어도 실행 전체가 얼어붙지 않습니다 — 새 연결이 그냥 우회합니다. (로컬 GPU 엔드포인트에서는 건너뜁니다. 새 연결을 열어도 같은 서버에 도착하기 때문입니다.)

- **병렬 분할 임베딩** — *하나*의 막힌 항목이 전체를 멈추게 하는 거대한 일괄 POST(555초짜리 몬스터 배치를 실측) 대신, VectFox는 배치를 1개 항목 단위 요청으로 분할해 최대 16개 동시 웨이브로 발사합니다. 불량 항목 하나의 피해 범위는 정확히 한 항목뿐 — 그것이 단독으로 재시도하는 동안 나머지는 모두 통과합니다.

- **파이프라인화된 추출 → 삽입(오버랩)** — 배치 N의 벡터 삽입이 배치 N+1의 LLM 추출과 겹쳐 진행되므로, 단일 느린 단계에서 파이프라인이 놀지 않습니다(긴 채팅에서 실시간 약 35% 단축). 추출 윈도우가 겹쳐 있어 윈도우 경계에 걸친 이벤트도 놓치지 않으며, Chrome 재시작에도 살아남는 핑거프린트 캐시 덕분에 중간에 멈춘 실행은 처음부터 다시 하지 않고 멈춘 지점에서 그대로 재개됩니다.

이것들은 상승 작용합니다: 환경이 나쁠수록 — 스토리가 길수록, 연결이 불안정할수록, 프로바이더가 느릴수록 — 더 강하게 작동합니다. 벡터화는 멈춰 서는 대신 복구하여 끝까지 완료합니다.

### 📝 저장 전 요약
벡터 저장 전 필수 LLM 요약. OpenRouter와 로컬 vLLM 호환 엔드포인트를 지원합니다. 구성 가능한 프롬프트 템플릿.

### ⏯️ 더 나은 벡터화 제어
정지 버튼, 일시 중지/재개, 중간 실행 중 Chrome 재시작에서도 살아남는 핑거프린트 캐시.

### 🔒 채팅별 컬렉션 범위
새 컬렉션은 현재 채팅에 자동 활성화됩니다. "Active for current chat" 체크박스가 채팅 잠금을 제어합니다. Database Browser의 잠금 버튼은 잠금이 *이* 채팅용인지 다른 채팅용인지 표시합니다.

### 📡 더 똑똑한 상태 표시기
Auto-Sync 카드는 현재 채팅이 벡터화되었는지와 얼마나 많은 이벤트가 존재하는지 보여줍니다. World Info 카드는 벡터화된 로어북을 이름으로 보여줍니다. 누락된 것이 있으면 둘 다 올바른 벡터화 도구로 링크됩니다.

### 🗂️ 탭 인터페이스
설정은 **Core**(백엔드, 임베딩, 하이브리드), **EventBase**(채팅 이력/아카이브 채팅 파일 .jsonl), **ChunkBase**(로어북/문서/URL/wiki), **Action**(진단, 개발 도구)으로 나뉩니다.

### ⚡ 병렬 윈도우 — 벡터화 가속
긴 채팅을 벡터화하는 것은 보통 한 번에 한 윈도우씩 처리합니다: 윈도우 1을 LLM에 보내고 → 기다리고 → 임베딩하고 → 윈도우 2 보내고 → 기다리고 → 임베딩하고 → ... 2000 메시지 채팅의 경우 이것은 많은 직렬 대기입니다.

**병렬 윈도우** 슬라이더(Vectorize Content의 Chunking Strategy 섹션)는 최대 **8개의 LLM 추출 + 임베딩 호출을 동시에** 띄울 수 있게 합니다. 윈도우 1이 추출되는 동안 윈도우 2–8도 진행 중이어서 총 수집 시간을 극적으로 단축합니다.

| 슬라이더 값 | 동작 |
|---|---|
| **1 (안전)** | 한 번에 한 윈도우. 가장 낮은 제공자 부하, 속도 제한 위험 없음, 가장 느림. |
| **2–4** | 가벼운 병렬. 대부분의 제공자에 좋은 중간점. |
| **5–8 (빠름)** | 공격적 병렬. 속도 제한이 높은 클라우드 제공자(OpenRouter, OpenAI, vLLM)에 최적. 무료 티어에서 속도 제한에 걸릴 수 있음. |

엄격한 속도 제한 무료 티어나 단일 로컬 GPU에 있다면 **1**을 사용. 유료 클라우드 제공자에 있고 2000 메시지 채팅을 한 시간이 아닌 몇 분 안에 수집하고 싶다면 **8**까지 올리세요.

### 🧹 다국어 키워드 품질
CJK를 위한 더 나은 단일 문자 필터링, 고신호 1자 RPG/일상생활/학교 용어를 위한 모드별 예외.

### 🧹 대규모 정리
혼합 백엔드 검색, handle ID 필터링 주변의 수많은 버그 수정 및 원본 VectHare로부터의 기타 향상.

## 🎭 활성화 규칙

각 컬렉션 카드에는 활성화 패널이 있습니다. 우선순위 체인은:

1. **비활성화** (일시 정지 버튼) → 절대 쿼리하지 않음
2. **트리거** → 키워드가 최근 메시지와 일치 → 활성화
3. **고급 조건** → 트리거가 비어있거나 일치하지 않으면 조건 규칙 평가 → 활성화
4. **현재 채팅에 활성 / 캐릭터 잠금** → 수동 항상 켜기 폴백
5. **아무것도 일치하지 않음** → 활성화하지 않음

조건은 감정(Character Expressions 스프라이트 감지를 통해), 키워드, 메시지/턴 카운트, 결합된 AND/OR 규칙을 지원합니다.

> ⚠️ **레거시 기능 참고:** 트리거 및 조건 기반 활성화는 **VectHare에서 상속받은 기능**이며 하위 호환성을 위해서만 유지됩니다. VectFox의 주요 아키텍처 변경으로 인해, **사용자는 항상 컬렉션을 "현재 채팅에 활성"으로 설정하거나 캐릭터 잠금을 사용해야 하며**, 트리거/조건에 의존하지 말아야 합니다. 벡터 검색의 핵심은 검색 엔진이 모든 것을 검색할 수 있게 하는 것입니다—키워드 트리거 기반의 선택적 활성화는 그 목적을 훼손합니다.
> 
> **CJK 참고:** 트리거와 감정/키워드 조건도 **영어 전용**입니다—키워드 사전이 영어이고 regex의 `\b` 단어 경계는 CJK 문자 사이에서 발동하지 않습니다. 중국어/일본어/한국어 스토리의 경우 대신 **"현재 채팅에 활성" / 캐릭터 잠금**을 사용하세요. Message Count / Turn Count 조건은 숫자이며 모든 언어에서 잘 작동합니다.

```bash
Windows에서는 명령 프롬프트, Linux/Mac에서는 터미널, docker에 있다면 콘솔에 들어가세요
```

### 단계 3: VectFox 설정
1. **VectFox Settings** 열기 (익스텐션 패널의 Core 탭).
2. 벡터 저장소 선택 (Standard 또는 Qdrant).
3. 임베딩 제공자 선택 (Transformers, vLLM, Ollama, OpenRouter, etc.).
   - 💡 **권장:** **OpenRouter**를 통해 `qwen/qwen3-embedding-8b`를 사용하세요. 매우 저렴하고 다국어 지원이 뛰어나며 (CJK 및 라틴 문자 모두 우수) VectFox가 대상으로 하는 코퍼스 규모에 대해 고품질 밀집 벡터를 생성합니다.
4. 요약 LLM 선택 (OpenRouter 또는 vLLM) — 벡터화 중 EventBase 추출에 사용됩니다.
   - 💡 **추천 OpenRouter 모델:** 저렴하고 빠른 추출 경로로는 `openai/gpt-4o-mini` 또는 `google/gemini-3.1-flash-lite`를 사용하세요 — 둘 다 비용과 수집 지연을 낮게 유지합니다. 더 높은 추출 품질을 원하고 비용을 더 지불해도 괜찮다면, `x-ai/grok-4.3`이 더 강력하지만 더 비싼 선택지입니다. OpenRouter가 오래된 모델 ID(예: `x-ai/grok-4.1-fast`)에 대해 404 또는 사용 중단 오류를 반환한다면 해당 ID는 피하세요. 모델 가용성은 시간이 지나며 바뀔 수 있으므로, 긴 수집 작업을 실행하기 전에 OpenRouter에서 정확한 모델 ID를 확인하세요. **Agent Mode LLM** (AgentMode 탭에서 별도 설정)에도 같은 권장이 적용됩니다 — AgentMode의 모델 필드를 비워두면 이 요약기 설정을 상속합니다.
5. 클라우드 제공자(OpenRouter / vLLM )를 사용한다면 API 키를 구성합니다.
6. **Keyword Extraction**에서 스토리의 언어를 선택합니다.
7. 대부분의 설정은 기본값으로 두면 됩니다 — 자유롭게 조정하세요.
8. SillyTavern에서 채팅을 클릭한 다음 VectFox 익스텐션 아이콘을 다시 클릭하세요. 첫 DB를 벡터화하려면 "Vectorize Content"를 클릭하고 **Chat History**를 선택**해야 합니다**.
9. 필요시 **AutoSync** 탭에서 Auto-Sync를 활성화합니다. 빈도는 EventBase 탭 아래 *Extraction > Window Size*에서 제어합니다.
10. 필요시 **WorldInfo** 탭에서 로어북 / World Info를 벡터화합니다.
11. (선택) 다른 설정이 모두 작동하면 **AgentMode** 탭에서 **Agent Mode**를 켤 수 있습니다. provider / model / API key를 비워두면 요약기 설정을 상속합니다 — 즉 단계 4에서 사용한 저렴하고 빠른 모델이 플래너도 구동하게 됩니다. 자세한 작동 방식은 위의 "작동 방식 → Agent Mode" 섹션을 참조하세요.

## ❓ FAQ

**왜 내부적으로 두 개의 별도 파이프라인이 있나요?**
내부적으로 VectFox는 콘텐츠 유형에 따라 두 가지 검색 경로 중 하나로 콘텐츠를 라우팅합니다:

| 파이프라인 | 처리하는 것 |
|---|---|
| **EventBase** | 채팅 이력(라이브 채팅 + 업로드된 `.jsonl` 아카이브) |
| **Standard (Chunk)** | 그 밖의 모든 것: 로어북, 캐릭터 카드, URL, 문서, wiki 페이지, YouTube 자막 |

서로의 콘텐츠를 절대 보지 않습니다—그래서 같은 채팅 메시지가 두 번 검색되지 않습니다(한 번은 이벤트로, 한 번은 원시 청크로). 보통 이것을 생각할 필요는 없습니다; 그저 EventBase가 채팅을 소유하고, 표준 청크 파이프라인이 나머지 모든 것을 소유한다는 것을 의미할 뿐입니다.

**채팅 중간에 CJK 토크나이저 모드를 변경할 수 있나요?**
아니요—하지 마세요. CJK 토크나이저 모드는 센티넬 포인트를 통해 upsert 시 **각 Qdrant 컬렉션에 고정**됩니다. 이미 콘텐츠를 벡터화한 후 모드를 전환하면 다음 쿼리에서 "Tokenizer mismatch" 경고 모달이 발동되며, 실제 옵션은 다음뿐입니다:
1. 원래 모드로 **되돌리기**(기존 벡터 보존), 또는
2. 새 모드로 **컬렉션을 처음부터 다시 벡터화**(추출된 모든 이벤트를 버리고 처음부터 시작).

채팅을 벡터화하기 *전에* 토크나이저 모드를 선택하고 그것을 고수하세요. 인플레이스 마이그레이션은 없습니다—스파스 벡터는 원래 모드로 토큰화되었기에 다른 토크나이저의 출력과 호환되지 않습니다. 채팅을 벡터화하기 *전에* 임베딩 모델도 선택하고 고수하세요.

**중국어/일본어/한국어 채팅에서 트리거/감정 조건을 사용할 수 있나요?**
신뢰할 수 없습니다. 키워드 사전은 영어 전용이고 regex의 `\b` 단어 경계는 CJK 문자 사이에서 발동하지 않습니다. "현재 채팅에 활성" / 캐릭터 잠금을 사용하거나 숫자 Message Count / Turn Count 조건을 사용하세요.
**컬렉션을 안전하게 삭제하려면 어떻게 해야 하나요?**
**항상 VectFox 내부의 Database Browser를 사용하세요**(Action 탭 → "Database Browser" 버튼). 컬렉션 카드의 빨간 "Delete" 버튼을 클릭하세요. 이렇게 하면 벡터 데이터베이스**와** VectFox의 내부 레지스트리 + 메타데이터를 모두 올바르게 정리합니다. Qdrant 웹 UI에서 수동으로 컬렉션을 삭제하거나 SillyTavern의 `settings.json`을 편집해서 삭제하지 마세요—그렇게 하면 고아 메타데이터가 남아 동기화 충돌, "collection not found" 오류 및 기타 이상한 동작을 일으킵니다。
**"scene" 설정을 찾을 수 없는 이유는? 원본 VectHare에 있었는데요**
Scene 지원은 제거되었습니다(청크 기반 채팅 시대의 기능이었고, 채팅은 이제 EventBase를 통해 실행됩니다). 이벤트를 함께 그룹화하는 것은 그다지 논리적이지 않아 기능이 제거되었습니다.

**시스템 설명이 너무 기술적이어서 이해할 수 없어요!**
동의합니다. 이 확장 기능은 일반적인 SillyTavern 채팅에는 과할 수 있습니다. 도대체 누가 10,000번 이상 답장하는 채팅을 하겠습니까? 하지만 검색 품질은 중요합니다.

**이 확장 기능은 멀티유저나 공개 배포 환경에서 안전한가요?**
VectFox + Similharity는 **개인 사용을 위해 설계되었습니다** — 자신의 컴퓨터 또는 신뢰할 수 있는 사설 네트워크(LAN) 환경에서의 사용을 전제로 하며, 인터넷 공개 환경이나 멀티유저 공유 환경에는 적합하지 않습니다. 두 가지 이유가 있습니다:

- **Qdrant**는 기본 설치 시 사용자별 접근 제어 기능을 제공하지 않습니다. 동일한 Qdrant 인스턴스를 공유하는 경우, 해당 포트에 접근할 수 있는 누구나 모든 컬렉션을 읽고 쓸 수 있습니다. 두 사용자가 같은 Qdrant 서버를 공유하면 서로의 VectFox 컬렉션을 읽고 수정할 수 있습니다.
- **Similharity 플러그인**에도 사용자별 인증 기능이 없습니다. 백엔드(OpenAI, Ollama, vLLM 등)로의 임베딩 및 쿼리 요청은 사용자 수준의 격리 없이 전달됩니다.

Qdrant에 RBAC를 추가하고 플러그인에 사용자별 ACL을 구현하는 것은 드롭인 벡터 메모리 확장의 범위를 크게 벗어나는 대규모 작업이므로 지원 범위 밖입니다. 멀티유저 격리가 필요한 경우, 사용자별로 독립된 Qdrant 인스턴스와 SillyTavern을 실행하거나 컨테이너화된 배포 방식을 사용하세요.

참고: SillyTavern 자체는 `enableUserAccounts = true`를 통해 사용자별 데이터 격리를 지원합니다. 이 기능은 ST 자체 데이터(채팅, 설정, 캐릭터)를 사용자별로 분리하지만, 공유 Qdrant 서버에는 적용되지 않습니다.

**벡터화 중에 긴 타임아웃이나 "500" 오류가 계속 발생합니다. 어떻게 해야 하나요?**
**"Group embedding calls(임베딩 호출 묶기)" 체크를 해제해 보세요**(Core 탭 → Embedding 섹션).

기본적으로 VectFox는 여러 텍스트를 묶어 **하나의** 요청으로 임베딩 제공자에게 보냅니다. 모든 것이 정상일 때는 이 방식이 더 빠르고 저렴합니다. 하지만 제공자나 그 앞단의 게이트웨이(OpenRouter, 클라우드 릴레이 등)가 상태가 나쁘면, 그 하나의 큰 요청이 멈추거나 `500`으로 돌아올 수 있고, 그러면 **배치 전체가 함께 실패**합니다.

체크를 해제하면 VectFox는 **아이템당 하나의 요청**으로 전환됩니다. 이제 어떤 아이템이 멈춘 서버 워커에 걸리더라도 영향을 받는 것은 그 아이템 하나뿐이고 나머지는 통과합니다. 네트워크 통신은 조금 늘어나지만, 불안정한 업스트림과 함께 가라앉는 대신 그것을 **우회**합니다.

> 💡 평소에는 **체크된 상태로** 두세요. 배치 전체 타임아웃이나 `500`이 보이기 시작하면 **체크를 해제**하여 응급 처치로 삼고 다시 실행하세요.

**"Hedge slow embedding calls(느린 임베딩 호출 헤지)"는 무엇이고 필요한가요?**
**멈춘** 임베딩 요청에 대한 자동 안전장치입니다. **기본적으로 켜져 있으며(15초)** 대부분의 사용자는 건드릴 필요가 없습니다.

이것이 해결하는 문제는 이렇습니다. 때때로 클라우드 임베딩 제공자에 대한 요청이 실패하는 것이 아니라 그냥 **멈춥니다**. 연결이 멈춘 워커로 라우팅되어 아무것도 하지 않고 머물러 있다가, SillyTavern이 약 2분 후에 마침내 포기할 때까지 그대로인 경우가 많습니다. 이 하나의 멈춘 요청 때문에 벡터화 전체가 이유 없이 멈춰버립니다.

헤지가 이를 해결합니다. 요청이 제한 시간(15초) 내에 응답하지 않으면, VectFox는 첫 번째 요청을 취소하지 않고 **새 연결로 두 번째 동일한 요청**을 조용히 보냅니다. 먼저 응답하는 쪽이 이기고, 진 쪽은 버려집니다. 새 연결은 보통 *정상* 워커로 라우팅되므로, 전체 타임아웃을 기다리는 대신 몇 초 만에 복구됩니다. (같은 텍스트를 두 번 보내도 무해합니다. 중복은 동일한 데이터로 데이터베이스의 같은 항목을 덮어쓸 뿐입니다.)

> 💡 켜 두세요. 이미 무언가 잘못되고 있을 때만 작동하며, 불안정한 클라우드 제공자의 고통을 크게 줄여줍니다. **로컬** 모델(Ollama, Transformers, llama.cpp, KoboldCpp)에서는 자동으로 건너뜁니다(새 연결을 만들어도 라우팅이 바뀌지 않기 때문). `0`으로 설정하면 비활성화됩니다.

## 🐛 문제 해결

**"No embeddings available"** — Vectorize Content 아이콘을 클릭하고, 메인 ST 설정에서 Vectors 확장 기능을 활성화한 후, 임베딩 제공자를 선택하고, 필요 시 API 키를 추가한 다음 Diagnostics를 실행하세요.

**이벤트/청크가 검색되지 않음** — SillyTavern 로비가 아닌, 이미 채팅 안에 들어와 있는지 확인하세요.
Database Browser 아이콘을 클릭 → 해당 채팅의 컬렉션을 클릭 → 컬렉션이 "Active for current chat"으로 되어 있는지 확인하세요.

**"이 메시지에서 AI가 실제로 무엇을 회상할까?"** — Actions 탭의 **Debug Query** 버튼을 사용해 "가정" 테스트를 해보세요. 텍스트를 입력하면 실제 채팅 메시지를 보내지 않고도 라이브 데이터베이스에서 어떤 이벤트가 검색될지와 해당 점수를 정확히 확인할 수 있습니다. 임계값 조정과 중요한 이벤트가 올바르게 인덱싱되었는지 확인하는 데 유용합니다.

**"Backend health check failed"** — Qdrant에서 Qdrant 서버가 실행 중이고 Similharity 플러그인이 설치되었는지 확인하세요.

**느린 성능** — Qdrant + A3로 전환(단일 왕복, 서버측 융합). EventBase Top K 줄이기. 로컬 GPU(직렬) 대신 API 임베딩 제공자(병렬) 사용.

**대규모 채팅 기록의 벡터화가 느림** — VectFox는 기본적으로 API 레이트 제한을 적용합니다(초당 60회 호출). 무료 티어 백엔드를 보호하기 위해 벡터화 속도를 높이려면: **설정 → Core 탭 → Embedding → API Rate Limiting → \"Max Calls\"를 0으로 설정하여 비활성화**. 프로덕션 Qdrant의 경우 서버 용량에 따라 100+ 호출/분으로 증가할 수 있습니다.
