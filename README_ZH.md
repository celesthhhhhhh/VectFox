## 🎯 VectFox 是什麼？

以優秀的 VectHare 為基礎打造，VectFox 是**專為 SillyTavern 設計的高效能長期記憶系統**。它遠遠超越了單純的多語言 fork — 所有智慧檢索邏輯皆在真實向量資料庫（Qdrant）的伺服器端執行，結構化的事件導向方式取代了原始的摘要分塊，大幅提升召回精準度。即使超過 2,000 則訊息，查詢結果仍可在 3 秒內返回。支援所有語言；英文、日文、韓文、繁體/簡體中文另有停用詞精調。

我之所以分支原本的 VectHare，是因為我自己的 [MVU Game Maker](https://github.com/KritBlade/MVU_Game_Maker) 專案有非常大規模的需求。這類專案的特性包括：
- **極端規模：每篇故事 2,000+ 則回覆，每則回覆 1,000+ 字。摘要檢索的回應時間不到 3 秒**
- 支援所有語言。英文、日文、韓文、繁體/簡體中文另附停用詞列表精調；西班牙文、德文、阿拉伯文、印地文等語言亦可直接使用。
- 自動移除 [MVU Game Maker](https://github.com/KritBlade/MVU_Game_Maker) 的所有功能性標籤。
- 真正可用的超長期記憶（2000+ 則訊息）

一般的 SillyTavern 記憶擴充元件在這種負載下會完全撐不住，尤其是故事裡含有大量 [MVU Game Maker](https://github.com/KritBlade/MVU_Game_Maker) 使用的功能性標籤時更是如此——這些標籤對記憶查詢沒有任何幫助。因此，我需要一套工具，能在大規模資料下清理這些標籤，同時仍然維持高速向量化。

大多數記憶擴充元件都是為 100 則訊息以內的對話設計的，在那種規模下確實運作得不錯。但只要對話量超過這個門檻，它們就會被迫越來越激進地壓縮舊訊息。到最後，你會得到近期歷史的完整細節，再加上一團被壓得模糊不清的舊內容——而且這個問題基本上無法靠調整解決，因為你根本不可能把 100+ 則原始訊息塞進提示詞，或塞進自動建立的世界書條目裡。舊記憶**一定**會被壓縮，而這就代表細節一定會流失。

方輪汽車不管怎麼調校，都不會突然變好開。我需要的是適合這件事的正確工具。真正需要的，是一個專用的向量資料庫後端，能把所有記憶妥善保存下來。

***我決定為 SillyTavern 打造一個架構正確、接近生產環境設計的記憶系統。讓我們把記憶做到極致！***

因此，VectFox 使用專用向量資料庫來儲存聊天中的**每一個有意義的事件**。不管是第 1 則訊息，還是第 2,000 則訊息，每個有意義的事件都會保留在資料庫裡，讓 SillyTavern 隨時可以搜尋。我想做的是一套生產等級的 SillyTavern 記憶向量系統，能擴展到 10k+ 則訊息，並在數秒內完成完整往返搜尋。

### 它解決了哪些問題

- 🧠 **原本的 VectHare 不會思考什麼值得記住。** VectFox 在上面加了一層由 LLM 驅動的 **EventBase** 萃取：AI 自己判斷哪些片段是「有意義的事件」，並為每個事件標上相關的人物、物品、地點與概念。
- 🤖 **原本的 VectHare 不會推理你問了什麼**——它只做表面的文字相似度比對。VectFox 加入可選的 **Agent Mode**，用一個小型 LLM 規劃多角度搜尋，把你原本查詢字面上找不到的記憶也撈出來。
- 😩 在記憶儲存前，移除 [MVU Game Maker](https://github.com/KritBlade/MVU_Game_Maker) 使用的所有功能性標籤。
- 🧠 在 [MVU Game Maker](https://github.com/KritBlade/MVU_Game_Maker) 的角色記憶之上，加入以故事為基礎的記憶。
- 💸 漫長對話會讓你的 token 預算被無關的歷史內容塞爆。
- ✍️ 你不再需要手動編輯上下文來提醒角色記住關鍵事件。
- 🎯 **垃圾進，垃圾出。** 將原始聊天摘要成非結構化的文字塊後向量化，會產生雜訊高、訊號低的檢索結果。EventBase 強制使用**嚴格的結構化格式**（`characters`、`items`、`locations`、`concepts`、`keywords`、`importance`、`DateTime`），並以向量資料庫的原生格式儲存——而不只是文字塊。Agent Mode 接著用**定向結構化查詢**（依角色、概念、地點篩選）來查詢這些高度結構化的資料，大幅提高命中率，遠勝於對非結構化摘要做單純的文字相似度比對。

**VectFox 的解法：** 使用 **Qdrant** 作為專用向量資料庫，儲存聊天中的每個有意義事件，不管聊天長到什麼程度都能保留。對於還不想額外執行服務的使用者，**只使用 A1 與 A2 路徑的「輕量版」** 也可以直接跑在 SillyTavern 內建的向量儲存上，不需要安裝額外軟體——它與完整向量資料庫版本共用許多功能，只是可承載的規模較小。等你準備好建立真正的長期記憶系統時，再升級到使用 Qdrant 的 **A3 路徑**。

注意：Qdrant 是免費且開源的。

### VectFox 不打算解決的問題

VectFox 是一套**記憶系統**，不是追蹤系統。它不會追蹤任務進度、角色數值或即時世界狀態。對於這類需求，請搭配 [MVU Game Maker](https://github.com/KritBlade/MVU_Game_Maker) —— 它是一套以角色為中心的追蹤系統，內建 GUI 可管理任務、角色與數值。兩者一起運作大致能覆蓋長篇 SillyTavern 角色扮演中約 90% 的記憶與狀態問題。

## 🧠 它是怎麼運作的

向量搜尋就像一個非常聰明的「尋找」功能。它不是比對一模一樣的字，而是比對**意思**——所以你輸入「我餓了」，也能找到「我們去吃午餐吧」這類訊息，因為兩者的*意思*接近。

VectFox 建立在兩個互相配合的核心概念上：

### 1. EventBase — 從時間窗萃取事件，而不是替每則回覆做摘要

大多數記憶擴充元件會把每一則（或每幾則）AI 回覆摘要成一段文字。乍聽之下很合理，直到你仔細看回覆裡實際包含了什麼：

- 一段 100 句的回覆可能包含**5 個有意義的事件**（一場戰鬥、一個發現、一個約定、一次物品交換、一個關係轉折），但它們埋在 95 句對白填充、場景描寫與閒聊裡。
- 另一則回覆可能**完全沒有事件**——只是單純閒聊。
- 第三則回覆可能用 1000 字只講了**1 個事件**。

用「每則回覆做摘要」的方式，會把這三種情況全部壓平成「每則回覆一段摘要」——事件邊界會消失，重要轉折會跟填充內容混在一起，而且不管實際發生了什麼，最後都產出同一種形狀的資料。

**EventBase 會檢視一段近期訊息形成的時間窗，根據實際發生的內容萃取出 0、1 或多個結構化事件。** 每個事件都是一筆獨立紀錄，並附帶豐富的後設資料（`characters`、`locations`、`items`、`concepts`、`importance`、`DateTime` 等）——它不只是一串描述文字。

假設 Tav 與 Astarion 在另外 3 位隊員和背景故事雜訊之間，進行了一段長達 100 句的購物對話，EventBase 可能會從這堆文字中萃取出這樣一個事件：

```
{
event_type:    item_acquired
importance:    6
text:          塔芙與阿斯特里昂在博德之門採購了盔甲。阿斯特里昂嘲諷了那些價格。
               塔芙以80gp購買了一件皮革護胸。
DateTime:      1492-08-15T14:00:00
cause:         塔芙在夏爾試煉遠征前需要更好的盔甲
result:        塔芙現在穿著皮革護胸；隊伍資金花費了80gp
characters:    [塔芙, 阿斯特里昂]
locations:     [博德之門, 魔法雜貨店地區]
factions:      []
items:         [皮革護胸, 80gp]
concepts:      [盔甲採購, 隊伍經濟]
keywords:      [盔甲, 皮革, 護胸, 金幣, 購物]
open_threads:  [夏爾試煉的準備]

}
```

之後當你提到「還記得那次購物之旅嗎？」時，VectFox 會檢索**這個事件**——不是 100 句原文，也不是把購物之旅和旁邊無關閒聊平均混成一團的模糊摘要。

### 2. 為什麼專用向量資料庫是很自然的選擇

如果你只是替每則回覆做摘要，其實**不需要**向量資料庫。你產出的資料大致就是每則回覆一段、形狀始終差不多，然後隨著聊天成長不斷遞迴重新摘要。這種情況用一個簡單文字檔就夠了——這也是為什麼許多舊式記憶擴充元件從來沒有認真使用真正資料庫的原因。

但只要你採用 EventBase，情況就完全不同：

- 2,000 則回覆 → 可能產生 **1,000–3,000 個結構化事件**（有些回覆會萃取出好幾個事件，有些則一個都沒有）
- 每個事件都有豐富欄位：characters、locations、items、concepts、keywords、importance、timestamps
- 你必須在使用者對話進行中，即時透過**意義 + 關鍵字 + 後設資料過濾**，找出最相關的 5–10 個事件

這正是**Qdrant 這類專用向量資料庫原本就適合處理的工作負載**：大量帶有結構欄位的小型紀錄，同時支援密集向量相似度、稀疏關鍵字搜尋、後設資料過濾，以及跨完整語料的全域 BM25 權重。如果用一個扁平摘要檔案來做這件事，就會變成線性掃描、沒有關鍵字索引、沒有後設資料過濾，也無法擴展到幾百筆條目以上。

EventBase 並沒有*強迫*你使用 Qdrant——A1/A2 輕量路徑可以跑在 SillyTavern 內建的 Vectra 上。但只要你的聊天事件超過數百個，Qdrant 才是真正符合這種資料形狀的儲存層。

### 3. Agent Mode —— 讓 AI 幫你規劃搜尋（選用，僅限 Qdrant / A3）

純粹的向量搜尋有個弱點：它只能搜尋你字面上輸入的東西。當你問 *「你記得我當時為甚麼為你贖身嗎?」*，搜尋會去找匹配這些字的事件。但真正的答案可能牽涉到綁架（贖身的前因）、贖金談判（金額怎麼來的）、以及你角色後續的情緒反應——這些都不在你字面提到的內容裡。

**Agent Mode 加入一個小型「規劃師」LLM**，它會讀你最近的聊天紀錄和最佳語意命中結果，然後思考：*「我還應該從哪些角度搜尋，才能真正回答這個問題？」* 它會輸出 1–4 個後續搜尋查詢，平行送往 Qdrant。

**實際範例。** 你輸入：
> *我對 Mayla 説 "你記得我當時為甚麼為你贖身嗎?"*

不開啟 Agent Mode 時，搜尋會找到：
- ✓ 贖身付款事件本身（直接匹配「贖身」關鍵字）

開啟 Agent Mode 時，規劃師會自動加入這些角度（實際從這個查詢產出的規劃師輸出）：

```json
{
  "queries": [
    "Critblade 贖身 Mayla 的原因",
    "Mayla 恐懼被拋棄的過去",
    "Critblade 承諾幫助 Mayla 找父親",
    "Mayla 領導能力的挑戰"
  ],
  "filters": {
    "characters_any": ["Critblade", "Mayla"],
    "concepts_any": ["贖身", "恐懼拋棄", "領導力"]
  }
}
```

四個平行的 Qdrant 搜尋會回傳：
- ✓ 贖身付款（直接）
- ✓ Mayla 過去害怕被拋棄的情感背景
- ✓ 承諾幫她找尋失蹤父親的時刻（規劃師從最近聊天上下文推論出的相關劇情）
- ✓ 領導能力受到挑戰的橋段（規劃師拉進的相關線索）

四組結果與原本的搜尋合併，再透過同一個 4 權重重新排序器計算最終分數。主回覆 LLM 因此能看到**完整的因果鏈與情緒背景**，而不只是付款的那一瞬間。

**為什麼 Agent Mode 搭配 A3（Qdrant）**：
- 每個規劃師查詢都是獨立的 Qdrant 呼叫。Qdrant 並行處理 4 個查詢只需 1–3 秒。
- AgentMode 需要 Qdrant——在標準後端會完全跳過（記錄 `mode=SKIPPED reason=requires_qdrant_backend`）並直接返回預搜尋結果。沒有優雅退化，這是設計上的決定。
- Qdrant 的 payload 過濾器（`characters_any`、`locations_any`、`factions_any`、`items_any`、`concepts_any`、`event_type_any`、`importance_gte`）讓規劃師能精準縮小每次搜尋範圍——標準後端沒這個功能。

**成本與延遲**：使用 GPT-4o-mini 或 Haiku 等小模型作為規劃師，每輪約 $0.0002，並增加約 2–5 秒。它是**純加法**——絕不取代正常搜尋，且任何失敗都會乾淨地回退到標準流程。可在專屬的 **AgentMode** 頁籤設定；預設為關閉。

> 💡 Agent Mode 最適合用在**反思型問題**（「為什麼……」「還記得……嗎？」）、**模糊回憶**（「那次關於我父親的事」），以及**因果鏈查詢**——也就是字面訊息不含搜尋錨點的情況。如果你的訊息本身已經帶有正確的關鍵字，標準預搜尋通常就足夠了，這時 Agent Mode 是錦上添花。

### 🧠 為什麼它比傳統記憶擴充元件更好

大多數現有記憶擴充元件會採用以下兩種方法之一。而這兩種方法都會在聊天變長後失去細節。原因如下，也能看出 EventBase 是怎麼避開這個問題的：

| 面向 | 📝 滾動摘要 <br>*（多數「記憶」擴充元件）* | ✂️ 原始切塊 <br>*（舊式向量 RAG）* | 🧬 EventBase <br>*（VectFox）* |
|---|---|---|---|
| **儲存什麼** | 一段不斷成長的摘要文字 | 把每則訊息切成原始片段 | 帶後設資料的結構化事件紀錄 |
| **第 100 則訊息時** | 大致完整 | 完整 | 完整 |
| **第 200 則訊息時** | 嚴重壓縮——名字、數字、一次性細節開始漂移或消失 | Token 預算爆掉——舊片段會依分數被修剪或丟棄 | **完整**——舊事件仍在 DB 中，相關時才浮現 |
| **第 1,000+ 則訊息時** | 實際上已經變成一團模糊摘要 | DB 膨脹；檢索變得吵雜，因為原始片段訊號太低 | **完整**——只拉出與目前場景相關的少數事件 |
| **「壓縮」做了什麼** | 把摘要再摘要，每次都流失資訊 | 沒有壓縮——但也沒有合成；原始文字檢索很吃運氣 | 一次性、語意式處理——萃取*有意義的事件*並丟掉填充內容。事件本身的細節會被保留下來。 |
| **檢索訊號** | 無——整段摘要每一輪都會注入 | 對原始文字做向量相似度（能抓到同義改寫，但也很吵） | 對豐富欄位（`characters`、`items`、`locations`、`concepts`、`keywords` 加上密集語意）做向量 + BM25 混合搜尋 |
| **細節去了哪裡** | 壓縮後就永遠流失 | 片段分數低於門檻就流失 | **哪裡都沒去**——事件保存在向量資料庫裡，相關時就浮現 |
| **注入什麼** | 運作中的整段摘要（每一輪、每一次） | 幾則語意相近的原始訊息 | 只注入與當下訊息相關的事件 |

**核心洞見：** 滾動摘要會失去細節，是因為它為了騰出空間而*丟棄*舊內容。原始切塊會失去細節，是因為*檢索在規模變大後會崩壞*。EventBase 會永久保留每個有意義的事件——再由向量 + 關鍵字搜尋決定當下哪 5–10 個事件值得展示給 AI。細節沒有被壓縮掉；**被過濾掉的是無關內容**。

> 💡 **你的訊息措辭會大幅影響檢索結果。** 因為檢索是由你回覆的文字驅動，所以你選用的詞很重要。例如，*「Mayla，妳記得我為什麼付贖金嗎？」* 和 *「Mayla，妳記得我為什麼付了 2000 元嗎？」* 會回傳非常不同的事件——「贖金」會拉出所有與該故事線相關的事件（綁架、談判、交付），而「2000 元」主要只會匹配字面上提到 2000 這個數字的事件。若你想讓 AI 回想起特定場景，請用該場景中**有故事意義的字詞**來錨定你的訊息，而不是像精確數字這類附帶細節。
>
> 在 1,500 個事件的對話中做側對側比較：A3（Qdrant）對措辭良好的查詢把贖金事件排在 **#1 / #2**，對純數字查詢仍然把它們排在前面。A1 / A2（標準後端）**確實也找到了**同樣的贖金事件，但排名較低（措辭良好時約在 **#3 / #5** 左右），常常落在實際注入提示詞的事件範圍之外。這個差異是結構性的——A3 透過稀疏關鍵字索引搜尋完整語料，而 A1 / A2 只對密集向量層恰好撈到的候選做評分（請參閱下方路徑比較）。錨定措辭在所有後端上都很重要；只是 A3 在你猜錯詞時更為寬容。

## 🔍 混合搜尋：A1、A2 與 A3 路徑

VectFox 會結合**兩種訊號**來找出最佳結果：

- **訊號 1 - 向量相似度** — 根據意思匹配（「餓」可以匹配「去吃午餐」）
- **訊號 2 - BM25 關鍵字分數** — 精確字詞匹配（「Astarion」匹配「Astarion」）

依後端與設定不同，有**三條路徑**可以結合這兩種訊號：（從你的瀏覽器一路到 docker 上的專用向量資料庫）

### A1 — 標準後端 + BM25
VectFox 會先依「意義」找出大約 **100 個和你訊息*感覺*相關的事件**，再只針對這 100 個檢查你用到的確切字詞，最後把兩種分數合在一起。它輕量又快，適合較慢的電腦。

**但要注意：** 如果擁有完美關鍵字的那個事件沒被選進最初這 100 個，它就完全不會被看到——關鍵字這一步對它是隱形的。

<details>
<summary>⚙️ 技術細節</summary>

瀏覽器先執行向量搜尋，取回前 ~100 個候選結果，然後只針對這些候選結果計算 BM25 關鍵字分數。最後使用簡單的加權總和：`α × vectorScore + β × bm25Score`。

</details>

### A2 — 標準後端 + Hybrid（推薦給不使用 A3 路徑的大多數使用者）
第一步和 A1 一樣——抓出大約 100 個*感覺*相關的事件——但 A2 在**排序上更聰明**。它把同一批事件用兩種方式排名（一次看意義、一次看關鍵字），融合兩份列表，並對在*兩種*訊號上都表現好的事件給予加成。推薦給不走 A3 路徑的大多數使用者。

**但要注意：** 問題和 A1 一樣。A2 只能對它已經抓到的那 100 個事件重新排序——仍然無法把第一步就漏掉的關鍵字命中補進來。（只有 A3 能解決這點。）

<details>
<summary>⚙️ 技術細節</summary>

瀏覽器先執行向量搜尋取回前 ~100 個候選結果，然後對**同一個候選池**用兩種方式排名——一次用向量相似度，一次用 BM25——再用以下方式融合兩份排名列表：
- **RRF（Reciprocal Rank Fusion，倒數排名融合）** — 根據每份列表內的*排名位置*而不是原始分數來融合結果
- **雙訊號加成** — 同時出現在兩個列表中的結果最多可獲得 +8% 加成；只出現在單一列表中的結果則有小幅懲罰（僅向量 ×0.55 / 僅文字 ×0.60）

> ⚠️ **A2 並不是獨立的稀疏檢索。** 兩份排名列表都從*同一個密集 ANN 候選池*中取出。BM25 可以對密集層已經回傳的候選做重新排序，但無法把密集 ANN 漏掉的「純關鍵字命中」浮現出來。只有 **A3** 會為每個事件儲存真正的稀疏向量，並對整個語料執行稀疏檢索。

**範例：** 搜尋「Astarion drinks blood.」時，如果密集 ANN 因為「吸血鬼/飢餓」在語意上很接近而回傳了一個提到 Astarion 與血的事件，兩份排名都會把它浮現，雙訊號加成再把它往上推。如果某個稀有的「純關鍵字」事件落在前 100 個候選的窗口之外，A2 就找不到——但 A3 找得到。

**取捨（技術面）：** 在速度較快的電腦瀏覽器上能得到更好的融合效果，但召回仍然受限於密集 top-K（約 100 個候選）。

</details>

### A3 — Qdrant 原生稀疏 + 伺服器端 RRF + 公式重排（準確度最佳）
A3 是真正的大升級。它不再只先抓 ~100 個事件，而是讓 Qdrant **同時用兩種方式搜尋你的整個歷史**——一種看意義、一種看確切關鍵字——並在伺服器端用一次請求完成所有排名與過濾。所以即使是埋在 1,500 段對話之前某個事件裡的稀有字詞，也會被直接找到，而不會因為沒擠進第一批而被漏掉。

如果你使用 **AgentMode**，它還會更進一步：把你的問題拆成多個角度（*這是怎麼發生的？前因是什麼？還有誰參與？*），並在搜尋*之前*先用人物／地點／事物把範圍縮小，讓不相關的事件不會佔用名額。

**取捨：** 準確度最好，在大規模資料下速度也最快。需要一個 Qdrant 實例（免費、開源）。→ [Qdrant 安裝指南](Doc/Qdrant_install.md)

<details>
<summary>⚙️ 技術細節</summary>

A3 會把**整套流程在單一 API 呼叫中於 Qdrant 內部執行**。它相對於 A1 / A2 真正的結構性優勢不只是「更快」——而是 **A3 真的會用關鍵字搜尋整個完整語料**，而 A1 / A2 只用密集向量搜尋：

1. **針對「完整語料」的密集 + 稀疏混合檢索** — Qdrant 會在每個事件 upsert 時就在伺服器端建立一份稀疏關鍵字向量。查詢時，密集索引（意義）與稀疏索引（關鍵字）**會同時針對集合中的每一個事件平行搜尋**，再透過原生 RRF 融合兩份結果列表。這正是 A3 真正能夠規模化的關鍵：如果某個稀有關鍵字只出現在 1,500 段對話之前的一個冷門事件中，稀疏索引仍能直接找到它——完全不需要靠密集向量層恰好把它撈出來。（A1 / A2 只會對密集 ANN 撈出的 top-K 候選做評分，所以落在那扇窗外的事件不管關鍵字多匹配，仍然是看不見的。）BM25 IDF 也在伺服器端以全域方式計算，所以稀有字詞的評分一開始就是正確的。

2. **伺服器端公式重排** — Qdrant 接著在同一呼叫中，對那些混合結果套用 4 權重的重排公式。每個 `w_X` 都是使用者可調的權重滑桿；它所乘的項目都已正規化，讓權重可以乾淨地組合：`w_cosine × RRF_score + w_importance × (importance/10) + w_persist × (1 if should_persist else 0) + w_recency × exp_decay(source_window_end → chatLength)`。最終排名後的列表已排序完畢後才回傳。沒有額外的往返，瀏覽器 JavaScript 也不再做評分。

3. **伺服器端過濾** — 最低重要度門檻和上下文重複排除截止值（近期聊天中已可見的事件），在 Qdrant 內部就直接執行，而不是等結果到達後才處理。低於門檻的事件永遠不會離開伺服器。

4. **AgentMode 語意預過濾 + 多角度查詢** — AgentMode 會做兩件相互疊加的事。首先，規劃師 LLM 將使用者的問題分解為多個子查詢，從不同角度出發——不只是表面意思，還包括*這件事是怎麼發生的？*、*什麼導致了它？*、*後果是什麼？*、*還有誰參與其中？* 每個角度都作為獨立的向量搜尋執行。其次，規劃師會發出結構化的實體過濾器（`characters_any`、`locations_any`、`factions_any`、`items_any`、`concepts_any`、`event_type_any`、`importance_gte`），作為 Qdrant 有效負載子句在同一次呼叫中應用，在向量搜尋執行之前就縮小候選集合。這兩者的結合非常強大：多角度查詢透過語意撒下廣泛的網，而精確的過濾器則確保每次搜尋都鎖定在正確的實體範圍內——不相關的角色、地點或事件類型永遠不會佔用召回的名額。

瀏覽器只負責錨點加成（片語匹配）、成對去重，以及跨多個集合的最終合併。

**範例：** 搜尋「I cast Fireball at the dragon.」時，Qdrant 同時搜尋密集（呪術/攻擊意義）與稀疏（字面上的「Fireball」+「dragon」），透過 RRF 融合，依 importance/recency 公式排名，過濾低重要度事件——最後在一次呼叫中回傳最終的注入就緒列表。

**AgentMode 範例：** 詢問「我們和 Shadowheart 做了什麼交易？」規劃師會發出 `characters_any: ["Shadowheart"]`，並展開為多個子查詢：原始問題、*「涉及 Shadowheart 的協議或承諾」*、*「導致與 Shadowheart 交易的事件」*、*「Shadowheart 要求了什麼作為回報」*。每個子查詢都在已限縮於 Shadowheart 標籤事件的 Qdrant 候選集合中執行——廣泛的語意召回，零跨角色干擾。

</details>

## ✨ 功能特性

### 🧬 EventBase — 由 LLM 萃取的聊天記憶
LLM 會把聊天摘要成結構化事件，並附上重要性/新近性/持續性權重。一個 4 權重重新排序器會決定哪些事件要被注入。內建去重機制會抑制已經在近期訊息中可見的事件。

每個事件都是一筆結構化紀錄，而不是原始文字。以下是 LLM 從 Astarion 購物時間窗產出的範例：

```
{
event_type:    item_acquired
importance:    6
text:          塔芙與阿斯特里昂在博德之門採購了盔甲。阿斯特里昂嘲諷了那些價格。
               塔芙以80gp購買了一件皮革護胸。
DateTime:      1492-08-15T14:00:00
cause:         塔芙在夏爾試煉遠征前需要更好的盔甲
result:        塔芙現在穿著皮革護胸；隊伍資金花費了80gp
characters:    [塔芙, 阿斯特里昂]
locations:     [博德之門, 魔法雜貨店地區]
factions:      []
items:         [皮革護胸, 80gp]
concepts:      [盔甲採購, 隊伍經濟]
keywords:      [盔甲, 皮革, 護胸, 金幣, 購物]
open_threads:  [夏爾試煉的準備]

}
```

兩種訊號（意義 + 關鍵字）都會在這組豐富欄位上運作，所以像「armor for the dungeon」這樣的查詢會透過 concepts/open_threads 命中，而「Astarion 80gp」則會透過 characters/items/keywords 命中。這種結構是 Qdrant 向量資料庫的原生格式，因此命中率會比任何其他記憶擴充元件高得多。

### 🤖 Agent Mode —— 由 LLM 規劃的多角度檢索（僅限 Qdrant / A3）
一個小型規劃師 LLM 會讀你最近的聊天紀錄與前段預搜尋的候選結果，然後輸出 1–4 個後續 Qdrant 查詢，每個查詢都瞄準你問題的不同角度。結果並行擷取後合併，再透過同一個 4 權重重新排序器計算分數——**純加法，絕不取代正常搜尋**。任何失敗都會乾淨地回退到標準流程。最適合用在**反思型問題、模糊回憶、與因果鏈查詢**這些字面訊息不含搜尋錨點的場景。

- **提供者/模型繼承** —— AgentMode 頁籤的欄位若留空，就會自動繼承摘要器設定；也可以另外指定更便宜的模型（例如 Haiku 4.5、GPT-4o-mini）以節省規劃師成本。
- **語言匹配的提示詞** —— 規劃師會以聊天語言輸出查詢（中文、日文、韓文、拉丁語系、英文皆可），同時保留專有名詞原始形式。不會跨語言污染。
- **可調整滑桿** —— 送進規劃師的歷史輪數（1–10）、給規劃師看的候選數量（5–20）、最大查詢數（1–4）、逾時時間（1–60 秒）、除錯日誌。
- **實際成本** —— 用小型快速模型約每輪 $0.0002，多花 2–5 秒延遲。預設關閉——需要長篇回憶能力時再開啟。

請參考設定中的 **AgentMode** 頁籤，或上方「它是怎麼運作的 → Agent Mode」章節了解完整架構。

### 🌏 多語言支援

VectFox 可對任何語言進行分詞與索引——BM25/關鍵字處理完全語言中立。所有語言開箱即用，其中五種語言提供額外的停用詞精調。

| 語言 | 分詞器 | 停用詞列表 | 備註 |
|---|---|---|---|
| **英文** | Intl.Segmenter | ✅ 667 詞 | 永久啟用的基準線，適用於所有模式 |
| **日文** | TinySegmenter | ✅ 672 詞 | 去除「は・を・の…」等助詞 |
| **韓文** | Intl.Segmenter | ✅ 679 詞 | 去除「의・은・는…」等助詞 |
| **繁體中文** | Jieba WASM（TW 字典） | ✅ 899 詞 | 僅限 `jieba_tw` 模式，與簡體分開管理 |
| **簡體中文** | Jieba WASM | ✅ 994 詞 | 僅限 `jieba` 模式，與繁體分開管理 |
| **西班牙文、法文、德文、阿拉伯文、印地文…** | Intl.Segmenter | — | 可正常分詞與索引；無專屬列表，僅套用英文基準線 |

**停用詞列表採模式個別選擇，而非全語言聯集。** 韓文集合只參照英文＋韓文（約 1,346 詞），不再如過去那樣檢查 2,200 多個中文項目。這避免了漢韓共用漢字因意外符合中文助詞而被悄悄從索引中刪除的問題。

**結合符號文字**（印度系：印地文、泰米爾文、孟加拉文等；阿拉伯文的哈拉卡特等）以完整單詞為單位分詞——母音符號與結尾符（`्`）會被保留，`युद्ध`（「戰爭」）不會被拆解成碎片，而是完整索引。

**新增語言停用詞列表**只需一個 PR：加入單詞陣列 + 在 locale 登錄檔新增一行 + 在模式表新增一筆記錄。不需要修改其他任何地方。

- CJK 分詞器模式會在 upsert 時**鎖定於每個 Qdrant 集合**——切換模式時會顯示警告對話框

### 🛡️ 堅韌的向量化 — 無論如何都會跑完

長篇故事 × 糟糕的連線，正是樸素向量化工具陣亡的地方：一次卡住的嵌入呼叫、一次飄忽的逾時，就能讓 2,000 則訊息的匯入在半途崩潰。VectFox 的設計讓向量化**永遠向前推進、永遠能夠復原**——三道各自獨立的安全網，分別封住一種不同的故障模式。

- **請求對沖（hedge）** — 當一次嵌入呼叫超過門檻（預設 15 秒）仍無聲無息時，VectFox 不會傻等，而是在一條全新的連線上發出一份*複製*請求。最多 3 個對沖與原始請求賽跑，最先回應者勝出，落後者直接丟棄。單一卡住的 SiliconFlow 工作節點、或單一糟糕的 OpenRouter 路由決策，都不再凍結整個流程——新連線會直接繞過它。（本機 GPU 端點會跳過此機制，因為新連線仍會落到同一台伺服器上。）

- **並行拆分嵌入** — 不再用一個巨大的批次 POST（*單一*卡住的項目就能拖垮全部——我們實測到 555 秒的怪獸批次），VectFox 會把批次拆成單項目請求，以最多 16 個並行的波次發出。任何壞項目的波及範圍剛好就是那一個項目——它獨自重試的同時，其餘全部順利通過。

- **管線化的抽取 → 插入（重疊）** — 批次 N 的向量插入與批次 N+1 的 LLM 抽取重疊進行，管線不會因為單一慢階段而閒置（長對話的實際耗時約縮短 35%）。抽取視窗彼此重疊，跨越視窗邊界的事件也不會漏掉；而能在 Chrome 重啟後存活的指紋快取，讓中途停下的工作不必從頭來過，直接從停下的地方續跑。

這些機制會相互疊加：環境越差——故事越長、連線越不穩、供應商越慢——它們就越賣力。向量化會復原並跑完，而不是卡死在半路。

### 🔍 原生稀疏向量混合搜尋（Qdrant）
也就是上面提到的 A3 路徑——伺服器端 RRF 加上全域準確的 BM25 IDF。每次查詢只需要一次往返。

### 📝 儲存前先摘要
強制在向量儲存前先進行 LLM 摘要。支援 OpenRouter 與本地 vLLM 相容端點。提示詞範本可自行設定。

### ⏯️ 更好的向量化控制
提供停止按鈕、暫停/恢復，以及即使 Chrome 在執行中重啟也能保留下來的指紋快取。

### 🔒 以每個聊天為範圍的集合
新集合會自動為目前聊天啟用。「為當前聊天啟用」核取方塊會控制聊天鎖定。資料庫瀏覽器中的鎖定按鈕會顯示鎖定是針對*這個*聊天，還是針對另一個聊天。

### 📡 更聰明的狀態指示器
Auto-Sync 卡片會顯示目前聊天是否已向量化，以及有多少事件。World Info 卡片會以名稱顯示已向量化的世界書。如果有缺漏，兩者都會連結到正確的向量化器。

### 🗂️ 分頁介面
設定分為 **Core**（後端、嵌入、混合）、**EventBase**（聊天歷史/封存 .jsonl 聊天檔）、**ChunkBase**（世界書/文件/URL/wiki）、**Action**（診斷、開發工具）。

### ⚡ 平行視窗 — 加速向量化
向量化一段長聊天時，通常會一次處理一個視窗：把視窗 1 送給 LLM → 等待 → 嵌入 → 送視窗 2 → 等待 → 嵌入 → …對於 2000 則訊息的聊天來說，這會產生大量串列等待時間。

**平行視窗**滑桿（Vectorize Content 內的 Chunking Strategy 區段）最多可以讓你同時啟動 **8 個 LLM 萃取 + 嵌入呼叫**。當視窗 1 正在萃取時，視窗 2–8 也能同時進行，大幅縮短總攝取時間。

| 滑桿值 | 行為 |
|---|---|
| **1（安全）** | 一次一個視窗。對供應商負載最低，沒有速率限制風險，但最慢。 |
| **2–4** | 輕度平行。對多數供應商來說是很好的折衷。 |
| **5–8（快速）** | 積極平行。最適合速率限制較高的雲端供應商（OpenRouter、OpenAI、vLLM）。可能觸發免費方案的速率限制。 |

如果你使用的是速率限制很嚴格的免費方案，或只有單一本地 GPU，請使用 **1**。如果你使用付費雲端供應商，並想把 2000 則訊息的聊天從一小時縮短到幾分鐘內攝取完成，請開到 **8**。

### 🧹 多語言關鍵字品質
CJK 的單字元過濾更完善，並針對高訊號的單字 RPG/日常生活/校園詞彙，提供依模式區分的例外規則。

### 🧹 大規模清理
修復了許多混合後端搜尋與 handle ID 過濾方面的錯誤，也包含來自原始 VectHare 的其他改進。

## 🎭 啟用規則

每個集合卡片都有一個啟用面板。優先順序如下：

1. **停用**（暫停按鈕）→ 永不查詢
2. **觸發詞** → 關鍵字匹配近期訊息 → 啟用
3. **進階條件** → 如果觸發詞為空/沒有匹配，就評估條件規則 → 啟用
4. **為當前聊天啟用 / 角色鎖定** → 手動永遠啟用的後備方案
5. **沒有匹配** → 不啟用

條件支援關鍵字/模式、發言者、出場角色、訊息/輪數計數、時段，以及組合式 AND/OR 規則。

> ⚠️ **舊有功能說明：** 觸發詞與條件式啟用功能**繼承自 VectHare**，保留於此僅為向後相容。由於 VectFox 的架構有重大變更，**使用者應該一律將集合設為「為當前聊天啟用」或使用角色鎖定**，而不要依賴觸發詞/條件。向量搜尋的重點就是讓搜尋引擎搜尋所有內容——基於關鍵字觸發詞的選擇性啟用會破壞這個目的。
> 
> **CJK 注意：** 觸發詞與條件式啟用是**語言中立**的——關鍵字/模式比對使用純粹的子字串搜尋，對中文/日文/韓文與英文一樣有效。（只有你自己撰寫、使用 ASCII `\b` 詞邊界的 regex 不會在 CJK 字元之間觸發——請改用純子字串，或使用 `\p{L}` 搭配 `u` 旗標。）

## 💾 安裝

### 步驟 1：安裝擴充元件

1. 在瀏覽器中開啟 SillyTavern
2. 前往 **Extensions** 面板（拼圖圖示）
3. 點擊 **"Install Extension"**
4. 貼上這個 URL：
   ```
   https://github.com/KritBlade/VectFox
   ```
5. 點擊 **Install**

就這樣！VectFox 會自動下載並啟用。

### 步驟 2：安裝 Similharity 插件（Qdrant 必要 — Standard 可選但建議安裝）

**使用 Qdrant 後端時必須安裝。** 使用 Standard 後端時為可選，但安裝後即使在 Standard 上也能解鎖額外功能：

| 功能 | Standard（無插件） | Standard（有插件） | **Qdrant（Cloud / Local）** ⭐ |
|---|---|---|---|
| 事件搜尋與注入 | ✅ 完整功能 | ✅ 完整功能 | ✅ 完整功能 |
| 嵌入與向量化 | ✅ 正常運作 | ✅ 正常運作 | ✅ 正常運作 |
| 關鍵字儲存於 DB（提升搜尋） | ❌ 僅儲存於本機設定 | ✅ 儲存於向量 DB | ✅ 儲存於向量 DB |
| 事件重要性分數 | ❌ 遺失（原生 API 會捨棄元資料） | ⚠️ 已儲存，但未用於重新排序 | ✅ 已儲存並用於重新排序 |
| 在資料庫瀏覽器中檢視區塊 | ❌ 不支援 | ✅ 支援 | ✅ 支援 |
| 編輯個別區塊 | ❌ 不支援 | ✅ 支援 | ✅ 支援 |
| 搜尋演算法 | A1/A2 混合 | A1/A2 混合 | **A3 混合（準確度最佳）** |
| 擴展至大型資料集 | ⚠️ 長篇故事下效能下降 | ⚠️ 長篇故事下效能下降 | ✅ 1 萬+ 訊息仍保持快速 |
| 安裝設定 | 無需 | 安裝插件 | 安裝插件 + 免費雲端帳號 **或** [本機安裝](Doc/Qdrant_install.md) |

如果你使用 Standard 後端且未安裝插件，事件搜尋與注入仍可正常運作，只是無法進行區塊檢視或使用關鍵字元資料。若追求最佳的檢索品質，**強烈建議使用 Qdrant** —— https://qdrant.tech/ 的免費雲端方案完全不需要本機設定。

#### 插件安裝說明：

```bash
在 Windows 上開啟命令提示字元，或在 Linux/Mac 上開啟終端機；如果你是在 docker 上，請進入 Console
cd SillyTavern/plugins
git clone -b Similharity-Plugin https://github.com/KritBlade/VectFox.git similharity
cd similharity
npm install
```

在 `config.yaml` 中搜尋以下鍵並改為 true：（Windows 通常會在 SillyTavern\config.yaml，而 Linux/Mac 應該在 SillyTavern\config\config.yaml）
```yaml
enableServerPlugins: true
```
重啟 SillyTavern。

### 步驟 3：設定 VectFox
1. 開啟 **VectFox Settings**（擴充元件面板中的 Core 分頁）。
2. 選擇你的向量儲存（Standard 或 Qdrant）。
3. 選擇你的嵌入供應商 (Transformers, vLLM, Ollama, OpenRouter, etc.).
   - 💡 **推薦：** 透過 **OpenRouter** 使用 `qwen/qwen3-embedding-8b`。這個模型非常便宜，多語支援優秀（CJK 與拉丁語系都好），對 VectFox 這種規模的語料庫能產出高品質的密集向量。
4. 選擇你的摘要 LLM（OpenRouter 或 vLLM）—— 向量化時 EventBase 萃取會用到。
   - 💡 **推薦的 OpenRouter 模型：** 如果想要便宜又快速的萃取路徑，使用 `openai/gpt-4o-mini` 或 `google/gemini-3.1-flash-lite`——兩者都能把成本與向量化延遲維持在低點。如果你想要更高的萃取品質且不介意多付一些費用，`x-ai/grok-4.3` 是更強但更貴的選擇。如果 OpenRouter 對較舊的模型 ID（例如 `x-ai/grok-4.1-fast`）回傳 404 或棄用錯誤，請避免使用它；模型可用性會隨時間改變，因此在長時間向量化之前，請先在 OpenRouter 確認確切的模型 ID。**Agent Mode LLM**（在 AgentMode 分頁另外設定）也適用同樣的推薦——如果你把 AgentMode 的模型欄位留空，它會自動繼承這個摘要器設定。
5. 如果使用雲端供應商（OpenRouter / vLLM ），請設定 API 金鑰。
6. 在 **Keyword Extraction** 中選擇你故事使用的語言。
7. 大多數設定維持預設值即可，但你也可以依需求調整。
8. 在 SillyTavern 中進入你的聊天，然後再次點擊 VectFox 擴充元件圖示。你**必須**點擊 "Vectorize Content"，並選擇 **Chat History** 來向量化你的第一個 DB。
9. 如有需要，請在 **AutoSync** 分頁啟用 Auto-Sync；頻率則由 EventBase 分頁下的 *Extraction > Window Size* 控制。
10. 如有需要，請在 **WorldInfo** 分頁向量化你的世界書/World Info。
11. （選用）在其他設定都正常後，可以到 **AgentMode** 分頁開啟 **Agent Mode**。把 provider / model / API key 留空就會自動繼承你的摘要器設定，這樣步驟 4 中那個便宜快速的模型也會用來驅動規劃師。完整功能說明請見上方「它是怎麼運作的 → Agent Mode」章節。

## ❓ 常見問題

**為什麼底層有兩條獨立管線？**
在內部，VectFox 會根據內容類型，把內容路由到兩條檢索路徑之一：

| 管線 | 處理什麼 |
|---|---|
| **EventBase** | 你的聊天歷史（即時聊天 + 上傳的 `.jsonl` 封存） |
| **Standard（Chunk）** | 其他所有內容：世界書、角色卡、URL、文件、wiki 頁面、YouTube 字幕 |

它們永遠看不到彼此的內容——所以同一則聊天訊息不會被檢索兩次（一次作為事件，一次作為原始片段）。你通常不需要特別思考這件事；這代表 EventBase 負責聊天，而標準片段管線負責其他所有內容。

**我可以在聊天中途更改 CJK 分詞器模式嗎？**
不行——不要這樣做。CJK 分詞器模式會在 upsert 時透過哨兵點**鎖定於每個 Qdrant 集合**。如果你已經向量化內容後才切換模式，下一次查詢時就會觸發「分詞器不匹配」警告對話框。你真正能選的只有：
1. **還原**到原始模式（保留現有向量），或
2. **從頭重新向量化集合**（丟掉所有已萃取事件，從零開始）。

請在開始向量化聊天**之前**就選好分詞器模式，並且之後都不要更改。這裡沒有原地遷移——稀疏向量是用原始模式分詞出來的，所以和不同分詞器的輸出不相容。也請在開始向量化聊天**之前**選好你的嵌入模型，並且之後固定使用同一個模型。

**我可以在中文/日文/韓文聊天中使用觸發詞/條件嗎？**
可以。關鍵字/模式比對使用純粹的子字串搜尋，對 CJK 與英文一樣有效。唯一的例外是你自己撰寫、使用 ASCII `\b` 詞邊界的 regex（不會在 CJK 字元之間觸發）——請改用純子字串，或使用 Unicode `\p{L}` 搭配 `u` 旗標。數值條件（Message Count / Turn Count）則永遠與語言無關。

**為什麼找不到「scene」設定？原本的 VectHare 有這個功能**
Scene 支援已被移除（那是基於切塊聊天時代的功能，而現在聊天會透過 EventBase 處理）。把事件硬聚合在一起不太合理，所以這個功能被移除了。

**系統的說明聽起來太技術性了，我看不懂！**
我同意。這個擴充元件對於一般 SillyTavern 聊天來說可能是殺雞用牛刀了。到底誰會聊到 10,000+ 則回復啊？但檢索品質確實很重要。

**這個擴充元件適合多人使用或對外公開的部署環境嗎？**
VectFox + Similharity 是為**個人使用設計的** — 適合在自己的機器或受信任的私人區域網路（LAN）上使用，並不適合公開網際網路或多使用者共用環境。主要原因有兩個：

- **Qdrant** 在預設安裝下沒有使用者層級的存取控制。共用同一個 Qdrant 執行個體時，任何能連到該連接埠的人都可以讀寫所有集合。如果兩位使用者共用同一台 Qdrant 伺服器，他們就能互相讀寫對方的 VectFox 集合。
- **Similharity 插件** 也沒有使用者層級的驗證機制。它將嵌入與查詢請求轉發到你的後端（OpenAI、Ollama、vLLM 等），不具備使用者層級的隔離。

為 Qdrant 新增完整的 RBAC，以及為插件加入使用者層級的 ACL，已超出這個即插即用向量記憶體擴充元件的設計範疇。如果需要多使用者隔離，請為每位使用者分別執行獨立的 Qdrant 執行個體與 SillyTavern，或使用容器化部署方式。

請注意：SillyTavern 本身支援透過 `enableUserAccounts = true` 達到使用者層級的資料隔離——這個功能會把 ST 自己的資料（聊天記錄、設定、角色）按使用者分開儲存，但並不延伸到共用的 Qdrant 伺服器。

**向量化時總是遇到長時間逾時或「500」錯誤，怎麼辦？**
試試**取消勾選「Group embedding calls（合併嵌入呼叫）」**（Core 分頁 → Embedding 區段）。

預設情況下，VectFox 會把一整批文字打包成**一個**請求送給嵌入提供者——在一切正常時這樣更快也更省。但如果提供者或它前面的閘道（OpenRouter、雲端中繼等）狀態不佳，這一個大請求就可能卡住或回傳 `500`，於是**整批一起失敗**。

取消勾選後，VectFox 會切換為**每筆一個請求**。這樣即使某一筆撞上卡住的伺服器工作程序，受影響的也只有那一筆，其餘的照常通過。這會多一點網路往返，但能**繞開**不穩定的上游，而不是和它一起沉沒。

> 💡 平時請**保持勾選**。一旦開始出現整批逾時或 `500`，就**取消勾選**作為應急處理並重試。

**「Hedge slow embedding calls（對慢速嵌入呼叫進行對沖）」是做什麼的？我需要它嗎？**
它是針對**卡住**的嵌入請求的自動安全網。**預設開啟（15 秒）**，大多數人無需改動。

它解決的問題是：有時候發給雲端嵌入提供者的請求並不會失敗——它只是**卡住**。連線被路由到一個卡死的工作程序上，什麼也不做地乾等著，往往要到 SillyTavern 大約 2 分鐘後最終放棄為止。這一個卡住的請求會無緣無故地拖住整個向量化流程。

對沖（Hedge）解決了這個問題。如果一個請求在時限（15 秒）內沒有回應，VectFox 會在**不取消第一個請求**的前提下，悄悄用**一條新連線發出第二個完全相同的請求**。哪個先回傳就用哪個，輸的那個被丟棄。新連線通常會被路由到*健康*的工作程序，所以你能在幾秒內恢復，而不必乾等整個逾時。（把同一段文字發兩次是無害的——重複只會用相同的資料覆寫資料庫裡的同一筆記錄。）

> 💡 保持開啟即可。它只在事情已經出問題時才會觸發，能大幅減輕不穩定雲端提供者帶來的痛苦。對於**本機**模型（Ollama、Transformers、llama.cpp、KoboldCpp）會自動跳過（新建連線也改變不了路由）。設為 `0` 可關閉。

## 🐛 疑難排解

**"No embeddings available"** — 點擊Vectorize Content 圖標, 在 ST 主要設定中啟用 Vectors 擴充元件，選擇一個嵌入供應商，必要時加入 API 金鑰，然後執行 Diagnostics。

**事件/片段沒有被檢索到** — 確保您已經在聊天室內，而不是在 Sillytavern 的大廳。
點擊 Database Browser 圖示 → 點擊該聊天的 collection → 確認該 collection 已顯示為 "Active for current chat"。

**「這條訊息 AI 到底會召回什麼？」** — 使用 Actions 分頁中的 **Debug Query** 按鈕進行「假設情境」測試。輸入任意文字即可直接查詢正式資料庫，看到哪些事件會被檢索到及其分數，無需發送真實的聊天訊息。非常適合用來調整門檻，以及確認重要事件是否已正確建立索引。

**"Backend health check failed"** — 如果使用 Qdrant，請確認 Qdrant 伺服器正在執行，且已安裝 Similharity 插件。

**效能緩慢** — 切換到 Qdrant + A3（單次往返、伺服器端融合）。降低 EventBase Top K。使用 API 嵌入供應商（可平行）而不是本地 GPU（通常是串列）。

**大型聊天記錄的向量化速度緩慢** — VectFox 預設套用 API 速率限制（每 60 秒 60 次呼叫），以保護免費層後端。加快向量化速度：**設定 → Core 標籤 → Embedding → API Rate Limiting → 將「Max Calls」設為 0 以停用**。若要使用生產環境 Qdrant，可根據伺服器容量增加到 100+ 次呼叫/分鐘。
